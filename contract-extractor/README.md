# Contract Extractor

Два CLI-приложения для извлечения реквизитов сторон (арендодатель и арендатор)
из договоров аренды. На вход — текстовый файл с несколькими договорами,
разделёнными строкой-маркером (по умолчанию `=====`).

Оба приложения переиспользуют общее ядро [`../core`](../core) (клиент
OpenAI-совместимого API, конфигурация, типы) — без дублирования логики.

## Требования

- Node.js **>= 22.6**.
- Каталог `../core` рядом (импортируется напрямую).
- Свой `.env` в этой папке (см. `.env.example`): `LLM_API_KEY`, `LLM_BASE_URL`,
  `LLM_MODEL`. Переменные те же, что у `llm-cli`.

## Установка

```bash
npm install
cp .env.example .env   # затем впишите ключ/URL/модель
```

## Формат входного файла

```
ДОГОВОР АРЕНДЫ № 1
... текст первого договора ...
=====
ДОГОВОР АРЕНДЫ № 2
... текст второго договора ...
```

Разделитель можно переопределить флагом `--separator`.

## App 1 — строгий JSON (`extract-json`)

Извлекает реквизиты по каждому договору и печатает **массив объектов JSON**.
Использует все ограничения генерации: `json_schema` (строгая схема + дублирование
её текстом в промпте для провайдеров, которые игнорируют structured outputs),
`max_tokens` (предохранитель длины) и `stop` (страховка от «хвоста» после JSON).

```bash
npm run extract-json -- contracts.txt
npm run extract-json -- contracts.txt --limit=10 --batch=5 --max-tokens=2000 --no-thinking
```

| Флаг             | Назначение                                              | По умолчанию |
| ---------------- | ------------------------------------------------------- | ------------ |
| `--limit=N`      | Вернуть реквизиты только для первых N договоров         | все          |
| `--batch=K`      | Сколько договоров слать в одном запросе                 | 5            |
| `--separator=S`  | Строка-разделитель договоров в файле                    | `=====`      |
| `--max-tokens=M` | Лимит токенов ответа **на пакет**                       | 1500         |
| `--no-thinking`  | Отключить «рассуждения» модели (флаг без значения)      | выкл.        |

**Экономия токенов при лимите.** При `--limit=N` договоры обрабатываются
пакетами; как только набрано N объектов, оставшиеся пакеты **не отправляются**,
а на граничном пакете модель просят вернуть только недостающее число —
экономятся и входные, и выходные токены.

**Рассуждения и `--max-tokens` (важно для reasoning-моделей).** На reasoning-модели
(например, `glm-5.1`) лимит `--max-tokens` тратится в т.ч. на «размышления»: при
`--batch=2` и `--max-tokens=2000` весь бюджет может уйти на reasoning, и до JSON
дело не дойдёт — будет ошибка «не найден JSON-объект». Два решения:

- **`--no-thinking`** — отключить рассуждения. Извлечение реквизитов механическое,
  поэтому это безопасно: быстрее, дешевле и надёжно укладывается в небольшой лимит
  (рекомендуется для типовых договоров).
- либо **увеличить `--max-tokens`** (≈6000+ на пакет из 2), оставив рассуждения —
  если попадаются особо запутанные договоры.

## App 2 — человекочитаемый список (`list-text`)

Без ограничений формата: печатает реквизиты сторон обычным текстом, разделяя
договоры горизонтальной чертой.

```bash
npm run list-text -- contracts.txt
npm run list-text -- contracts.txt --batch=10 --separator="---"
```

| Флаг            | Назначение                              | По умолчанию |
| --------------- | --------------------------------------- | ------------ |
| `--batch=K`     | Сколько договоров слать в одном запросе | 5            |
| `--separator=S` | Строка-разделитель договоров в файле    | `=====`      |

## Структура

| Файл                       | Назначение                                                   |
| -------------------------- | ----------------------------------------------------------- |
| `src/contracts.ts`         | Разбиение файла на договоры и на пакеты                      |

| `src/extract-json.ts`      | App 1: оркестрация строгого режима + разбор аргументов       |
