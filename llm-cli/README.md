# LLM Console

Консольное приложение на Node.js + TypeScript для взаимодействия с любой LLM
через **OpenAI-совместимый** HTTP API chat/completions (запросы выполняются
нативным `fetch`). Привязки к конкретному провайдеру нет — подойдёт z.ai (GLM),
OpenAI, OpenRouter, локальный Ollama / LM Studio и любой другой совместимый
эндпоинт. Достаточно указать URL, модель и ключ.

## Требования

- Node.js **>= 22.6** (используется нативный запуск `.ts`-файлов и `fetch`).
- Доступ к OpenAI-совместимому API (URL, имя модели, API-ключ).

## Установка

```bash
npm install
```

## Настройка

Создайте свой файл `.env` в корне проекта — приложение автоматически подхватит
его при запуске. Проще всего скопировать шаблон и заполнить значения:

```bash
cp .env.example .env
```

Затем откройте `.env` и задайте обязательные переменные:

```bash
LLM_API_KEY=ваш_ключ
LLM_BASE_URL=https://api.z.ai/api/paas/v4/chat/completions
LLM_MODEL=glm-5.1
```

> Файл `.env` не коммитится (он в `.gitignore`) — храните в нём секреты спокойно.
> Переменные, уже заданные в окружении, имеют приоритет над значениями из `.env`.

Необязательные переменные: `LLM_TEMPERATURE`, `LLM_SYSTEM_PROMPT`,
`LLM_REQUEST_TIMEOUT_MS` (таймаут ожидания ответа модели в миллисекундах,
по умолчанию `60000` — увеличьте, если модель отвечает медленно),
`LLM_CONTEXT_TOKENS` (размер контекстного окна модели в токенах, по умолчанию
`8192`; задайте под свою модель), а также тарифы для оценки стоимости запроса:
`LLM_PRICE_INPUT_PER_1M` и `LLM_PRICE_OUTPUT_PER_1M` ($ за 1M токенов, по умолчанию
`0` — стоимость не считается) и `LLM_USD_RUB` (курс ₽/$, по умолчанию `90`).
`LLM_MAX_TOOL_ROUNDS` (потолок раундов агентного цикла чата с MCP-инструментами, по умолчанию
`20`). `LLM_EXECUTOR_MODEL` — модель роли **выполнения** пайплайна (напр. `qwen2.5-coder:7b`);
остальные роли идут на `LLM_MODEL`, не задан → всё на `LLM_MODEL`. Полезно, когда code-модель
сильнее на генерации кода, но слабее на планировании/проверке (замерено: coder даёт лучший код,
но как ЕДИНАЯ модель зацикливает проверку). На локали смена модели за роль стоит переключения в
Ollama, если модели не резидентны вместе (`OLLAMA_MAX_LOADED_MODELS`). `LLM_STRUCTURED_OUTPUTS=1` — включает генерацию этапов пайплайна строго по JSON-схеме
(constrained decoding). Полезно слабым локальным моделям, которые не держат формат; **по
умолчанию выключено**, так как `response_format` ломает z.ai/GLM. Пока флаг не поднят, поведение
байт-в-байт прежнее (JSON просим в промпте + толерантный парсер). Учтите: под схемой планировщик
не вызывает MCP-инструменты. Для голосового ввода `/voice` — `YANDEX_API_KEY` (+ опц. `YANDEX_FOLDER_ID`,
`YANDEX_STT_LANG`, `VOICE_INPUT_DEVICE`).

После каждого ответа в интерактивном чате печатается строка статистики:

```
[вход 37 · выход 4 · история ~48 · 5.3с · 64 ток/с · ≈ $0.000022 / 0.0020 ₽]
```

— входные и выходные токены запроса (точно, из `usage`), приблизительный размер
истории сессии в токенах, **время ответа и скорость генерации** (токены/сек — видна
разница между квантами/параметрами при ручном сравнении) и стоимость запроса в $ и ₽
(если заданы тарифы). Скорость печатается при последнем проходе генерации; ручное
сравнение конфигураций — в `docs/local-llm.md`.

При выходе из чата печатается **итоговая сводка за всю сессию** — суммарные
входные/выходные токены и общая стоимость:

```
Итого за сессию: вход 91 · выход 6 · всего 97 · ≈ $0.000068 / 0.0061 ₽
```

В интерактивном чате история диалога удерживается в пределах контекста модели.
Стратегия выбирается опцией `--memory` **в начале сессии** (по умолчанию `window`):

- **`window`** — скользящее окно по токенам: системное сообщение сохраняется
  всегда, бюджет = контекст (`--context-tokens` или `LLM_CONTEXT_TOKENS`) − резерв
  под ответ (`--max-tokens` или `1024`), и при переполнении из истории **выпадают**
  самые старые реплики.
- **`summary`** — сжатие: **последние N реплик** (`--keep-recent`, по умолчанию 6)
  хранятся дословно, а всё, что старше, **сворачивается в краткое системное резюме**
  (отдельным вызовом модели). Так давний контекст не теряется полностью. Каждый
  прогон сжатия печатается отдельной строкой `[сжатие · …]` и учитывается в итогах
  (это дополнительные токены). При сбое сжатия ход мягко откатывается к окну.
- **`facts`** — «липкие факты»: отдельный системный блок `ключ: значение` (цель,
  ограничения, предпочтения, решения, договорённости) **обновляется моделью после
  каждого хода** и отправляется вместе с **последними N репликами** (`--keep-recent`).
  В отличие от `summary`, факты обновляются всегда (а не только при переполнении) и
  хранят структурированные данные, а не пересказ; при противоречии **данные из
  реплик пользователя приоритетнее**, чем из ответов ассистента. Каждое обновление
  помечается строкой `[факты · …]` и идёт в итоги. При сбое обновления прежние факты
  сохраняются.

Оценка токенов грубая (нет провайдер-независимого токенизатора), поэтому считается
консервативно. Полный транскрипт всегда сохраняется в сессию — стратегия влияет
только на то, что уходит в модель.

Альтернатива `.env` — задать те же переменные прямо в окружении
(`export LLM_API_KEY=…`).

## Запуск

Интерактивный чат (с сохранением истории диалога):

```bash
npm start
```

Одиночный запрос (промпт передаётся аргументами):

```bash
npm start -- "Привет! Кто ты?"
```

Выход из интерактивного режима — команда `/exit` или Ctrl+C. Температуру можно
менять прямо в чате командой `/temp <число>` (например, `/temp 0.2`) — она
применится к последующим сообщениям.

Ответ модели выводится **потоком** по мере генерации. Пока reasoning-модель
«думает» (присылает только рассуждения), в терминале крутится индикатор
`думает…`; как только пойдёт видимый текст — индикатор гаснет и ответ печатается
по кусочкам. Индикатор показывается только в TTY (в пайпах/файлах его нет).
Потоковый вывод можно выключить флагом `--no-stream` (например, когда нужен
цельный ответ для строгого JSON).

В потоковом режиме `LLM_REQUEST_TIMEOUT_MS` работает как таймаут **простоя**: стрим
обрывается, только если новые данные не приходят дольше этого времени. Длинный, но
«живой» ответ не режется по общей длительности (в обычном режиме это, как и
раньше, таймаут на весь запрос).

> Допустимый диапазон температуры зависит от провайдера: z.ai/GLM принимает
> примерно `0–1`, OpenAI — `0–2`. Приложение проверяет только что число
> неотрицательное; слишком большое значение отклонит сам провайдер (HTTP 400).
> На фактологических вопросах и с включёнными рассуждениями эффект температуры
> почти не виден — заметнее всего на творческих запросах с `--no-thinking`.

### Ограничения генерации (флаги)

Запрос можно выполнять как со специальными ограничениями, так и без них —
они задаются флагами командной строки. Флаги действуют в обоих режимах
(в интерактивном — применяются к каждому сообщению):

| Флаг                  | Назначение                                                        |
| --------------------- | ----------------------------------------------------------------- |
| `--max-tokens=N`      | Жёсткий лимит на длину ответа (положительное целое)               |
| `--stop=SEQ`          | Стоп-последовательность; можно указать несколько раз              |
| `--json`              | Любой синтаксически валидный JSON (`response_format: json_object`)|
| `--json-schema=FILE`  | Строгий JSON по схеме из файла (`response_format: json_schema`)   |
| `--file=FILE`         | Вложить содержимое файла в запрос; можно указать несколько раз    |
| `--no-thinking`       | Отключить «рассуждения» модели (флаг без значения)                |
| `--no-stream`         | Выключить потоковый вывод (флаг без значения)                     |
| `--temperature=T`     | Температура модели (по умолчанию — из `LLM_TEMPERATURE`)          |
| `--context-tokens=N`  | Размер контекста для окна истории; приоритетнее `LLM_CONTEXT_TOKENS` |
| `--memory=window\|summary\|facts` | Стратегия памяти: окно (по умолчанию), сжатие или блок фактов |
| `--keep-recent=N`     | При summary/facts: сколько последних реплик хранить дословно |
| `--switch[=имя\|id]`  | Продолжить ветку: без значения — последнюю, иначе по имени/id     |
| `--branch=имя`        | Ответвиться в новую ветку с этим именем (база — `--switch` или последняя) |
| `--ephemeral`         | Не сохранять сессию на диск                                       |
| `--task=текст`        | Стартовать с активной задачей (слоистая память)                  |
| `--profile=имя`       | Активный профиль (персона); создаётся при отсутствии              |
| `--no-memory`         | Выключить слоистую память (профиль + задача)                     |
| `--profile-tokens=N`  | Размер блока профиля в токенах (иначе — эвристика от контекста)  |
| `--task-tokens=N`     | Размер блока задачи в токенах (иначе — эвристика от контекста)   |

Значение можно писать как `--flag=value` или `--flag value`. Примеры:

```bash
# одиночный запрос с лимитом и форматом JSON
npm start -- --max-tokens=200 --json "Верни {\"city\": ...} для Парижа"

# строгий JSON по схеме из файла
npm start -- --json-schema=./city.schema.json "Дай данные про Париж"

# отправить содержимое файла(ов) в запрос
npm start -- --file ./code.ts "Найди баг в этом коде"

# несколько стоп-последовательностей
npm start -- --stop="###" --stop="END" "Текст до маркера"

# интерактивный чат с лимитом длины на каждый ответ
npm start -- --max-tokens=300

# без рассуждений — быстрее/дешевле; на reasoning-моделях это ещё и
# делает --stop надёжным (иначе стоп-маркер тонет в «размышлениях»)
npm start -- --no-thinking --stop="###" "Сгенерируй абзац и заверши ###"

# одиночный запрос с заданной температурой
npm start -- --temperature=0.2 "Дай строгое определение рекурсии"

# без ограничений — как обычно
npm start -- "Привет!"
```

Файл для `--json-schema` содержит саму JSON Schema (объект); приложение
оборачивает её в строгий `response_format` (`strict: true`):

```json
{
  "type": "object",
  "properties": { "city": { "type": "string" }, "country": { "type": "string" } },
  "required": ["city", "country"],
  "additionalProperties": false
}
```

> **Поддержка зависит от провайдера — с мягкой деградацией.** `json_schema`
> (строгие structured outputs) гарантируется на провайдерах, которые его
> реализуют (например, OpenAI). Некоторые эндпоинты — в том числе z.ai/GLM-5.1
> на момент написания — принимают параметр, но **игнорируют** его (HTTP 200,
> без ошибки). На такой случай приложение **дублирует схему текстом в системном
> промпте** и просит вернуть строго соответствующий JSON. Благодаря этому
> `--json-schema` на практике даёт нужный формат и на игнорирующих провайдерах
> (проверено на GLM-5.1 — ответ приходит чистым JSON по схеме), хотя строгой
> гарантии на стороне API там нет.

## Сессии и ветки (история между запусками)

Каждый запуск интерактивного чата — **новая ветка** (по умолчанию `main`); она
сохраняется на диск **после первого обмена** (вопрос-ответ). По умолчанию история
между запусками не подхватывается автоматически — продолжение всегда явное.

Ветка — это сессия с осмысленным **именем** (`label`), по которому удобно
переключаться вместо длинного id. От любого места диалога можно **ответвиться**:
исходная ветка остаётся нетронутой, а продолжение идёт в новой. Так из одной точки
можно вырастить несколько независимых веток и свободно между ними переключаться.

```bash
npm start                          # новая ветка main
npm start -- --switch              # продолжить последнюю ветку
npm start -- --switch=<имя|id>     # продолжить конкретную ветку
npm start -- --switch=<имя> --branch=feature  # ответвиться от неё в ветку feature
npm start -- --branch=feature      # ответвиться от последней ветки в feature
npm start -- --ephemeral           # одноразовый запуск без сохранения
```

Ветки лежат в `~/.llm-cli/sessions/` (переопределяется переменной
`LLM_SESSION_DIR`), по файлу `<id>.json` на ветку. Хранится **полный транскрипт**
(включая системное сообщение, замороженное для воспроизводимости); в модель на
каждом ходу уходит обрезанное окно. Запись атомарная; повреждённый файл считается
отсутствующим. id ветки сортируемый по времени, поэтому `--switch` без значения
берёт последнюю. `--switch` принимает **и имя, и id**: сначала ищется ветка с таким
именем, иначе — по id (форматы не пересекаются).

Ветками можно управлять и **прямо в чате** (полный список — команда `/help`):

| Команда            | Действие                                              |
| ------------------ | ----------------------------------------------------- |
| `/sessions`        | список веток (имя, id, превью)                        |
| `/branch <имя>`    | ответвиться в новую ветку с именем от текущего места  |
| `/switch <имя\|id>` | переключиться на ветку по имени или id                |
| `/reset`           | начать новую пустую ветку                             |
| `/task [текст]`    | показать текущую задачу; с текстом — **установить** текущую задачу (исполнить — `/run`) |
| `/tasks`           | список задач                                          |
| `/task switch <id\|имя>` | переключиться на задачу (вернуться к старой)     |
| `/task done`       | закрыть текущую задачу                                |
| `/task delete <id\|имя,…>` | удалить задачу(и) (можно несколько через запятую) |
| `/run [id\|описание]` | исполнить задачу по этапам (текущую / по id / новую) |
| `/runs`            | список прогонов задач                                 |
| `/run status [id]` | статус текущего/указанного прогона                    |
| `/run continue [id]` | продолжить приостановленный прогон                  |
| `/run edit <правка>` | внести правку перед продолжением (сбрасывает счётчик проверок) |
| `/run abort`       | досрочно завершить прогон                             |
| `/invariants`      | список инвариантов (жёстких ограничений)              |
| `/invariant add <текст>` | добавить инвариант (нарушать нельзя)            |
| `/invariant delete <n,…>` | удалить инвариант(ы)                           |
| `/profile`         | показать активный профиль (что известно о вас)        |
| `/profiles`        | список профилей (персон), активный помечен `*`        |
| `/profile switch <имя>` | переключить/создать профиль (персону)            |
| `/profile rename <имя>` | переименовать активный профиль                   |
| `/profile delete <имя,…>` | удалить профиль(и) (активный → переключение на default) |
| `/profile forget <n,…>` | забыть пункт(ы) профиля (можно несколько через запятую) |
| `/system <текст>`  | сменить системный промпт (действует с этого момента)  |
| `/file <путь>`     | добавить содержимое файла в контекст диалога          |
| `/temp <число>`    | сменить температуру                                   |
| `/exit`, `/quit`   | выход                                                 |

## Слоистая память (персонализация)

Чтобы ответы были **конкретными под вас и под задачу**, а не общими, история
разделена на слои (включено по умолчанию; `--no-memory` отключает):

- **Кратковременная** — текущий диалог (стратегия `--memory`, см. выше).
- **Задачная** — цель, ограничения, решения и прогресс по текущей задаче;
  переживает сессии (`~/.llm-cli/tasks/`). Задача определяется **автоматически**:
  если реплика похожа на новую цель, агент предлагает «Сделать задачей сессии
  «…»? (да/нет)» и ставит её после подтверждения. Явно — `/task <текст>` или
  `--task`. Управление: `/task`, `/tasks`, `/task switch`, `/task done`. Ветки
  наследуют активную задачу.
- **Долговременная** — профиль пользователя (предпочтения, стек, стиль) в
  `~/.llm-cli/profiles/<имя>.json`. Можно держать **несколько профилей (персон)** и
  переключаться между ними; активна одна персона глобально (указатель
  `~/.llm-cli/profiles/.active`, по умолчанию `default`). Управление — `/profiles`,
  `/profile switch <имя>` (создаёт при отсутствии), `/profile rename <имя>`,
  `/profile delete <имя,…>` (удаление активного → переключение на `default`), флаг
  `--profile`. Старый
  одиночный `profile.json` автоматически переносится в профиль `default`.
  В профиль попадает **только то, что сказали вы сами** (или явно подтвердили) —
  предложения и допущения модели туда не записываются. Явные предпочтения
  добавляются сразу, устойчивые черты **консолидируются в конце сессии** из ваших
  реплик. Просмотр — `/profile`, удаление пунктов — `/profile forget <n,…>`.

Размеры блоков профиля и задачи — доли от контекста модели (с потолками),
переопределяются флагами `--profile-tokens` / `--task-tokens`. После каждого хода
**явно показывается, что и в какой слой записано** — напр. `[память] задача «…» ←
N фактов; профиль ← «…»` (или `без изменений`) и строка стоимости `[память · …]`;
консолидация в конце сессии — `[профиль] консолидировано: N пунктов` + `[профиль ·
…]`. Всё идёт в итоги. При `--ephemeral` слои работают в памяти, **на диск не
пишутся**.

## Инварианты (жёсткие ограничения)

Инварианты — это глобальные ограничения проекта, которые ассистент **не имеет права
нарушать**: выбранная архитектура, принятые техрешения, ограничения стека,
бизнес-правила. Хранятся **отдельно от диалога** в `~/.llm-cli/invariants.json`,
действуют во всех сессиях и задачах.

- Задаются явно — `/invariant add <текст>` — или **по авто-предложению**: если в
  реплике зафиксировано жёсткое решение, ассистент спросит «Сделать инвариантом «…»?
  (да/нет)» и добавит по подтверждению. Просмотр — `/invariants`, удаление —
  `/invariant delete <n,…>`.
- Ассистент **явно учитывает** их в рассуждениях: блок инвариантов и директива
  подмешиваются в контекст с высшим приоритетом.
- **Жёсткий контроль отдельным агентом-контролёром.** Вывод любого *решающего* агента
  (чат-ассистент и этапы пайплайна `planning`/`execution`/`completion`) проверяется
  контролёром на соответствие инвариантам. При нарушении контролёр **называет**
  нарушенный инвариант, и агент **перегенерирует** решение (до 2 раз). Если соблюсти
  не удалось — в чате ассистент **отказывается** выдавать решение (с названием
  инварианта), в `/run` этап встаёт на **паузу**. Когда инвариантов нет — контролёр не
  вызывается (нулевой оверхед). `--ephemeral` держит инварианты в памяти, без записи на диск.

## Исполнение задачи по этапам (`/run`)

Задача — единая сущность, а `/task` и `/run` — два действия над ней: «вести» и
«исполнить». Явная команда `/task <текст>` **только устанавливает** текущую задачу
(цель в память задачи) — исполнение пайплайном запускается **отдельной командой**
`/run` (без аргумента — текущую). А вот **автоопределённая** задача (по подтверждению
«Сделать задачей сессии «…»?») запускается сразу. Управлять прогоном можно командами
`/run …` (см. таблицу выше).

Фиксированный пайплайн (пропуск этапов запрещён, каждый даёт артефакт):

```
сбор требований → планирование → выполнение → проверка → завершение
```

- **Сбор требований** — аналитик ведёт уточнение **по одному вопросу**, выбирая
  следующий с учётом прошлых ответов, и предлагает к каждому вопросу ответ по
  умолчанию. Он **сам читает проект** (read-only инструменты git/rag): изучает
  релевантные файлы (README/package.json/конфиги/код) и **не переспрашивает** то, что
  выводится из репозитория, а задаёт только **решающие** вопросы (интент, объём,
  приоритеты, значимые крайние случаи) — либо завершает без вопросов, если всё уже
  определено. Пустой ответ принимает предложение; слово-стоп («стоп», «достаточно»,
  «хватит») завершает опрос досрочно; есть страховочный лимит вопросов. Ответы
  пишутся в память задачи и идут в контекст следующих этапов.
- **Планирование → выполнение → проверка** — придирчивый проверяющий сверяет
  результат с критериями приёмки. Провал → авто-возврат в выполнение (до
  `maxRetries`, по умолчанию **10**).
- При исчерпании лимита проверок прогон **не встаёт на паузу, а возвращается к сбору
  требований** (счётчик проверок сбрасывается, аналитику передаются замечания
  проверки) — до `maxRequirementCycles` циклов (по умолчанию **3**), затем пауза.
- **Завершение** требует подтверждения пользователя (`да` / `нет` / текст-правка).
  По подтверждению итог пишется в память задачи, задача помечается `done`.

**Ctrl+C** во время прогона ставит его на паузу на границе этапа (не выходит из
приложения); `/run edit <правка>` вносит правку и **сбрасывает счётчик проверок**;
`/run continue` продолжает с сохранённого этапа. `--ephemeral` держит прогоны в
памяти (без файлов и продолжения).

### Контролируемый жизненный цикл

Этапы — это явный автомат: переходы заданы таблицей разрешённых рёбер, а каждый этап
имеет предусловие (артефакт предыдущего). Сменить этап можно только допустимым
переходом — **перепрыгнуть этап нельзя** (нет выполнения без утверждённого плана, нет
завершения без пройденной проверки); недопустимый переход — это ошибка, а не молчаливый
пропуск. При продолжении прогона его состояние сверяется с артефактами: если файл
прогона оказался несогласованным (этап «впереди» своих артефактов), он **откатывается к
корректному этапу** и проходит недостающие шаги заново — об этом сообщается в консоль.

### Команда агентов на этап

Сложные задачи планирует **команда экспертов разных ролей**, а не один агент. Перед
планированием **оркестратор** (по сути — основной чат) по задаче решает, нужна ли команда
и какие роли (архитектор, безопасность, тестирование, UX, данные и т.п.). Простая задача
— один агент (как раньше); сложная — несколько ролей работают ограниченно-параллельно со
своих ракурсов, затем ведущий планировщик сводит их предложения в **единый план**. Состав
команды печатается строкой `🧩 Команда на этап …`, а вклады ролей отмечаются под планом и
сохраняются файлами-артефактами `planning-team-*`.

Управляется переменными окружения: `LLM_MAX_STAGE_AGENTS` — потолок ролей на этап (по
умолчанию **4**; `1` полностью выключает многоагентность, оркестратор не вызывается);
`LLM_STAGE_AGENT_CONCURRENCY` — максимум одновременных запросов роль-агентов (по умолчанию
**2**, бережно к ограничениям провайдера). Жёсткий контроль инвариантов применяется к
итоговому синтезированному плану. Многоагентность на этапе выполнения пока не включена.

## MCP-инструменты для агентов

Чат, а также агенты этапов **планирования** и **выполнения** могут вызывать инструменты
внешних **MCP-серверов** (function-calling). Серверы настраиваются в `~/.llm-cli/mcp.json`
(формат `mcpServers`, как у Claude Desktop) и подключаются на старте; флаг `--no-mcp`
выключает MCP. В чате при подключённых инструментах ход идёт без стриминга (агентный цикл).

Управление прямо из чата:

```
/mcp                                  список подключённых серверов и их инструментов
/mcp add <имя> <команда [аргументы]>  подключить stdio-сервер и сохранить в mcp.json
/mcp add <имя> <http(s)://…/mcp>      подключить Streamable HTTP сервер
/mcp remove <имя>                     отключить и убрать из конфигурации
/mcp reload                           переподключить всё из mcp.json
```

Пример: `/mcp add yandex node /путь/к/yandex-ocr-mcp/src/cli.ts` (ключи серверу удобнее
прописать в его секции `env` в `mcp.json`). Инструменты неймспейсятся как `сервер__инструмент`;
вызовы агентом печатаются строкой `🔧 инструмент …`. Требуется провайдер с поддержкой
function-calling (проверено на GLM и DeepSeek).

В репозитории есть готовые MCP-серверы для подключения: `yandex-ocr-mcp` (распознавание текста),
`scheduler-mcp` (планировщик), `filesystem-mcp` (файлы), `places-mcp` (поиск мест рядом),
`rag-mcp` (поиск по документам — см. ниже).

### Ответы по документам с источниками и цитатами (RAG, День 24)

При подключённом `rag-mcp` (`/mcp add rag node /путь/к/rag-mcp/src/cli.ts`) на вопросы о
репозитории/сайте/папке агент вызывает `search_docs` и отвечает **строго по найденным фрагментам** в
обязательном формате из трёх секций — **Ответ / Источники** (`source › section · chunk_id`) **/
Цитаты** (≥1 дословная выдержка-якорь). После ответа работает **локальный гейт против галлюцинаций**
(`citation-guard.ts`): источники должны быть среди реально найденных **и** есть **хотя бы одна
дословная цитата-якорь** (сверка строковая, подделать нельзя). Тело «Ответ» можно **синтезировать**
из нескольких фрагментов (процедура, порядок действий) — дословность всего ответа НЕ требуется, лишние
не-дословные записи в «Цитаты» терпим (иначе синтез-вопросы ложно падали в «не знаю»). Не проходит
(поддельный источник или нет ни одного якоря) — перегенерация (до 5 раз) с **адресным замечанием**,
иначе безопасный фолбэк. При пустом/слабом контексте (сигнал `⚠ Низкая уверенность` от сервера) — режим **«не знаю»**
с просьбой уточнить, без выдумок. Клиентские переменные (в окружении llm-cli, не в rag-mcp):
`RAG_ANSWER_COMPACT=1` — компактные источники/цитаты для длинного чата; `RAG_FAITHFULNESS_CHECK=1` —
опциональный LLM-судья достоверности поверх локального гейта (принимает обоснованный синтез, бракует
только противоречащее фрагментам или невыводимое). Ретрив/ранжирование настраиваются на
стороне сервера (`rag-mcp/.env`: `RAG_RERANK`/`RAG_REWRITE`/`RAG_CONFIDENCE_MIN`…).

> Гейт применяется только к **RAG-ходам** (когда реально вызван `search_docs`). Общий вопрос без
> адреса к документам агент может ответить из своих знаний, минуя гейт — это его решение.

### Grounded-мини-чат: `/rag` (День 25)

Чтобы каждый вопрос гарантированно шёл через базу знаний (а не по усмотрению модели), включите
grounded-режим:

```
/rag <источник1> [источник2 …]   привязать источники и включить режим (пишется в сессию, переживает перезапуск)
/rag                             показать привязанные источники
/rag off                         выключить
```

Тогда на **каждой содержательной** реплике (приветствия/благодарности/да-нет распознаются и идут
обычным ответом) llm-cli детерминированно ищет по **каждому** источнику, обогащая запрос **целью и
терминами из памяти задачи** (`/task`) — это держит цель диалога и чинит follow-up’ы («а подробнее?»).
Дальше ответ синтезируется **строго по найденным фрагментам, без вызова инструментов** (tool-free) +
гейт источников/цитат (см. выше). Tool-free важно для слабых локальных моделей: при полном агентном
цикле qwen 7B «игралась» с посторонними инструментами (например, планировала задачи в scheduler) вместо
ответа по докам. Так получается **мини-чат по документам с историей + памятью задачи + источниками**.
Нужны действия инструментами — выйдите из grounded (`/rag off`).

**Вопросы-воспоминания.** Реплики вроде «напомни, каким флагом…», «что мы решили», «с чего начали»
распознаются (гибрид: LLM-классификатор + лексические маркеры) и НЕ гонятся через новый поиск и цитатный
гейт — иначе на них приходил ложный отказ «не могу подтвердить». Вместо этого llm-cli **воспроизводит
прошлый ответ из истории максимально дословно, вместе с его секцией «Источники»** (temp=0). Если
подходящего ответа в истории нет — молча откатывается на обычный grounded-поиск (внутренний сигнал
пользователю не виден). Так ассистент не «глупеет» на отсылках к уже сказанному.

Проверено на 2 длинных диалогах (11 ходов): на маленьком корпусе — источники+цитаты на каждом ходе,
цель удерживается (финальный «напомни, с чего начали» отвечает верно); на большом реальном репозитории
(641 чанк) — большинство ходов с цитатами, остальные — честное «не знаю» (гейт отказывается вместо
выдумки). **Настройка под корпус** (в `rag-mcp/.env`): большой шумный корпус — `RAG_RERANK=llm` +
`RAG_REWRITE=hyde` (лучше recall, дороже); порог «не знаю» `RAG_CONFIDENCE_MIN` — компромисс: выше
строже к офф-топику, но чаще ложно отказывает на легитимных вопросах (ниже — наоборот). **Стоимость:**
каждый ход — поиск × N источников (+ rerank/rewrite на сервере) + гейт, так что за 10–15 ходов вызовов
заметно; для скорости на маленькой базе достаточно `RAG_RERANK=mmr` без hyde. За ход бывает несколько
поисков (принудительный + повторные агентом), их логи (`🔎 сводка` + `🔧 search_docs`) можно скрыть —
`RAG_QUIET=1` (в окружении llm-cli); ответ с секцией «Источники» при этом остаётся.

## Ассистент разработчика: проекты (`/project`, День 31)

Ассистент должен понимать **любой** проект, а не только тот, из которого запущен, и не один: фича
часто живёт в нескольких репозиториях сразу (бэкенд + фронтенд). Поэтому к сессии привязывается
**рабочее пространство проектов**.

```
/project                          привязанные проекты: корень, ветка, документация, команды
/project add <путь>               привязать локальный проект
/project add <git URL>            привязать удалённый — клонируется в кэш ~/.llm-cli/projects
/project remove <имя|путь>        отвязать один
/project off                      отвязать все
```

Привязка пишется в сессию (переживает перезапуск). Если не привязано **ничего**, а `llm-cli` запущен
внутри репозитория — проект определяется по текущему каталогу (сообщается строкой `📁 Проект …`);
как только привязан хотя бы один проект, автодетект больше не применяется — текущий репозиторий,
если он тоже нужен, добавьте явно.

**Привязка = разрешение.** `/project add` сам прописывает репозиторий в allow-list `git-mcp`
(переменная `GIT_ALLOWED_REPOS` в `mcp.json`) и переподключает сервер — иначе каждый вызов
инструмента (а за один `/ask` их 5–8) спрашивал бы подтверждение. Репозиторий, который **не**
привязан как проект, по-прежнему требует явного «да» (MCP elicitation) — гибкость не теряется.
`/project remove` и `/project off` **симметрично снимают** разрешение (убирают репозиторий из
`GIT_ALLOWED_REPOS` и переподключают сервер), возвращая конфигурацию в исходное состояние.
Трогается только то, что добавил клиент: репозитории, прописанные в `mcp.json` вручную, не
затрагиваются.

Что ассистент знает о проекте:

| Источник | Что даёт |
| -------- | -------- |
| Документация (`README`, `CLAUDE.md`, `docs/`, `openapi.*`, схемы) | Идёт в **RAG** — векторный поиск по смыслу |
| `git-mcp` (`git_branch`/`git_status`/`git_list_files`/`git_log`/`git_diff`/`git_grep`/`read_file`) | Структура, состояние, история и **сам код** — точечно, без индексации |
| `package.json` | Команды проекта: чем прогнать тесты, сборку, линтер |

Код в индекс НЕ кладётся сознательно: он большой, быстро устаревает в индексе и хорошо ищется
точным поиском (`git_grep`). Документация же выигрывает от поиска по смыслу.

Удалённый репозиторий сводится к локальному: клон `--filter=blob:none` (не `--depth=1` — неполная
история сделала бы `git log`/`git diff` бесполезными, а именно ими ассистент понимает проект).
Повторная привязка того же URL — `git fetch`, а не новый клон. Запрос логина git подавляется
(`GIT_TERMINAL_PROMPT=0`): недоступный репозиторий даёт внятную ошибку, а не молчаливое зависание.

Нестандартная раскладка документации — переменная `LLM_PROJECT_DOCS` (пути через запятую) заменяет
автоопределение.

### `/ask <вопрос>` — ответы о проекте с источниками

```
/ask какая структура у проекта?
/ask где обрабатывается авторизация?
/ask что изменилось в этой ветке?
```

Ход **полностью детерминированный** — инструменты вызывает сам `llm-cli`, а не модель:

1. Карточки проектов + живой git-контекст (ветка, изменённые файлы).
2. **Форс-поиск по документации** каждого проекта (`search_docs`).
3. Модель называет 1–3 **шаблона для grep** — узкая подзадача, посильная и слабой модели; выбирает
   она их по РЕАЛЬНОМУ списку файлов проекта (иначе транслитерирует русские слова и ищет мусор).
4. **Форс-`git_grep`** по шаблонам + **чтение файлов**: и тех, чьё имя названо шаблоном, и тех, куда
   указали совпадения (по одной строке grep код не понять, а файл-определение в выдачу может и не
   попасть — там будут лишь упоминания в импортах).
5. **Tool-free синтез** ответа и **цитатный гейт** (День 24): источники ⊂ найденного, ≥1 дословная
   цитата. Доказательствами служат и фрагменты документации, и **прочитанный код** — поэтому ответ о
   коде тоже проверяем, а выдумать его нельзя (сверка строковая).

Почему инструменты не отданы модели: агентный цикл на слабой локальной модели (`qwen2.5:7b`)
вырождался — она десятками раундов читала **выдуманный** файл `gates/citation_gate.txt`, пока не
кончался лимит раундов, и ответа не было вовсе. Тот же вывод, что и в grounded-режиме Дня 25.

Нет доказательств — честное «не знаю»; не сошлись цитаты — «не могу подтвердить», а не правдоподобная
выдумка. Нет привязанного проекта или не подключены `rag-mcp`/`git-mcp` — прямое сообщение об этом.

**Стоимость хода:** поиск × источники + вызов выбора шаблонов + grep × шаблоны + чтение до 3 файлов +
синтез (+ перегенерации гейта). На локальной 7B это минуты; на облачной модели — секунды.

### Проект в пайплайне (`/run`)

Привязанный проект виден агентам этапов — задача исполняется **в репозитории**, а не «в воздухе»:

| Этап | Что получает | Зачем |
| ---- | ------------ | ----- |
| Все этапы | Карточка проекта (корень, ветка, документация, команды тестов/сборки) | План и результат — про конкретный репозиторий |
| Планирование | + документация проекта (RAG) | План в терминах реальной структуры и соглашений |
| Проверка | + документация проекта (RAG) | Сверка результата с соглашениями проекта |
| Выполнение | Карточка + инструменты `git-mcp` | Код поверх существующего, в стиле проекта |

Документация подключается **адресно** — только там, где меняет результат: каждый поиск стоит
запросов, а на выполнении и завершении добор данных ничего не решает. Проект не привязан (или нет
`rag-mcp`) → пайплайн работает ровно как прежде.

### Работа с файлами и запуск команд в `/run` (День 34)

Если проект привязан, `/run` **реально правит файлы проекта и проверяет их запуском команд**, а не
только генерит текст. Цель задаётся на уровне намерения («обнови README под код», «зафиксируй ADR»,
«обнови зависимости») — ассистент **сам** решает, какие файлы прочитать/найти/создать/изменить.

Безопасно и воспроизводимо через **изолированную git-копию (worktree)**:

1. На прогон создаётся рабочая копия проекта от `HEAD` во временном каталоге — **рабочее дерево не
   трогается**.
2. **Выполнение** — исполнитель агентно (function-calling) читает/ищет/создаёт/изменяет/удаляет файлы
   **в копии** (инструменты `read_file`/`read_files`/`write_file`/`edit_file`/`delete_file`/`list_dir`/
   `grep` по путям относительно проекта; **`edit_file`** — точечная замена `old_string`→`new_string`
   (экономит выходные токены, предпочтительнее полной перезаписи `write_file`); `delete_file` — только
   файл, не каталог; служебные `node_modules`/`.git` недоступны). Если у проекта есть безопасные скрипты — доступен
   **`run_command`**: агент запускает скрипты проекта (форматтер/линтер/тесты) прямо в копии и
   **сам исправляет** свои правки до зелёного (деплой/lifecycle-скрипты не пускаются). Для задач
   **обновления зависимостей** — **`run_package_command`** (`npm/yarn/pnpm install/update`,
   `npx npm-check-updates`): агент реально поднимает версии командами менеджера; при первой такой
   команде копия получает **свою `node_modules`** (симлинк на реальный снимается) — установка не
   трогает реальный проект, в применение идут только `package.json`/lockfile.
3. **Проверка** — прогоняет **обнаруженные команды проекта** (`test`/`build`/`lint` из `package.json`)
   **в копии**, с подмешанными dev-переменными проекта (`.env`/`.env.development`/`.env.dev`/`.env.local`/…;
   оверрайд имён — `LLM_PROJECT_ENV_FILES`) — чтобы сборка/тесты видели нужное окружение, напр.
   `BUILD_PUBLIC_BASE`. Ненулевой код/таймаут → возврат в выполнение (агент
   правит), модель не зовётся; всё зелёное → результат идёт судье. Запускаются **только обнаруженные**
   команды (allow-list).
4. Перед подтверждением виден **diff** копии. По `да` правки **применяются к реальному проекту**
   (копированием изменённых файлов), копия удаляется. По `нет`/паузе копия цела для доработки;
   `/run continue` продолжает в той же копии.

Проект **не** привязан → файлы не трогаются, пайплайн работает как прежде (текстовый результат).
База копии — образ **рабочего дерева** (незакоммиченные правки трекнутых файлов видны через
`git stash create`), а не `HEAD`: пайплайн работает от актуального состояния (напр. видит уже
поднятую, но не закоммиченную версию зависимости). Ограничения: копия берётся для **первого**
привязанного проекта; `untracked`-файлы в копию не попадают.

### Оркестрация длинного флоу

Агент сам выбирает инструменты, маршрутизирует вызовы по серверам (через неймспейс) и связывает
их в цепочку, передавая результат одного на вход другому — в один заход через несколько серверов
(например: `get_my_location → places__find_places → filesystem__read_file →
filesystem__append_file → scheduler__schedule_task`). Длину агентного цикла задаёт
`LLM_MAX_TOOL_ROUNDS` (по умолчанию `20`). После хода печатается **трасса вызовов** —
`🧭 a → b → c` — видно выбор инструментов и порядок маршрутизации.

### Подтверждения операций (MCP elicitation)

Сервер может в рантайме запросить у пользователя разрешение на операцию (например `filesystem-mcp`
для пути вне песочницы). `llm-cli` показывает запрос строкой `⚠ … (да/нет)` и отправляет ответ
серверу. Без такого запроса операция вне разрешённой области отклоняется.

## Голосовой ввод (`/voice`)

Команда `/voice` записывает речь с микрофона (через `ffmpeg`, OggOpus; стоп — Enter), распознаёт её
в текст (**Yandex SpeechKit**) и **вставляет результат в строку ввода** — можно поправить и
отправить. Включается, если задан `YANDEX_API_KEY` (опц. `YANDEX_FOLDER_ID`, `YANDEX_STT_LANG`,
`VOICE_INPUT_DEVICE`) и запуск идёт в терминале. Нужны установленные `ffmpeg` и доступ к микрофону.

> Ответы ассистента в терминале рендерятся из markdown в ANSI (жирный, заголовки, инлайн-код) —
> в пайпах/файлах текст остаётся исходным markdown.

## Сборка в бинарник (Bun)

Приложение можно собрать в **один самодостаточный исполняемый файл** (без
необходимости Node/Bun на целевой машине) с помощью [Bun](https://bun.sh):

```bash
npm run build:binary        # → dist/llm-cli под текущую платформу
npm run build:binary:all    # → бинарники под linux/macos/windows в dist/
```

Под капотом — `bun build ./src/cli.ts --compile --minify --bytecode`. Кода это не
касается: нативный TS и web-API (`fetch`, SSE-стрим) работают как есть. Bun сам
подхватывает `.env` из рабочего каталога, так что рядом с бинарником нужен только
`.env` (или переменные окружения). Размер бинарника — около 60 МБ (внутри рантайм
Bun); старт — миллисекунды. Каталог `dist/` в `.gitignore` — артефакты не коммитим.

> Требуется установленный Bun: `curl -fsSL https://bun.sh/install | bash`.

## Проверка типов

Node запускает `.ts` нативно (срезает типы без проверки), поэтому сборка
не нужна. Типы проверяются отдельной командой:

```bash
npm run typecheck   # tsc --noEmit: ищет ошибки типов без генерации файлов
```

## Тесты

Unit-тесты используют встроенный тест-раннер Node (`node:test`) — без сторонних
зависимостей. Покрытие кода — 100% (строки, ветки, функции); пороги проверяются
самой командой и роняют её при падении.

```bash
npm test
```
