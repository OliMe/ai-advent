# Contract Generator

CLI-генератор больших списков **синтетических** договоров аренды. Пишет текст с
договорами, разделёнными строкой `=====`, — формат, который напрямую читает
[`../contract-extractor`](../contract-extractor) (замкнутый цикл «сгенерировал →
извлёк»). Переиспользует общее ядро [`../core`](../core).

## Требования

- Node.js **>= 22.6**.
- Каталог `../core` рядом (импортируется напрямую).
- Свой `.env` в этой папке (см. `.env.example`): `LLM_API_KEY`, `LLM_BASE_URL`,
  `LLM_MODEL`. Переменные те же, что у `llm-cli`.

## Установка

```bash
npm install
cp .env.example .env   # затем впишите ключ/URL/модель
```

## Запуск

```bash
# 10 договоров в stdout (значения по умолчанию)
npm run generate

# 100 договоров в файл, по 10 за запрос (быстро)
npm run generate -- --count=100 --per-request=10 --out=contracts.txt
```

| Флаг               | Назначение                                             | По умолчанию    |
| ------------------ | ------------------------------------------------------ | --------------- |
| `--count=N`        | Сколько договоров сгенерировать всего                  | 10              |
| `--per-request=K`  | Сколько договоров просить за один запрос               | 1               |
| `--max-tokens=M`   | Лимит токенов **на один договор**                      | 600             |
| `--concurrency=C`  | Сколько запросов слать параллельно                     | 2               |
| `--model=NAME`     | Модель генерации                                       | `glm-4.5-flash` |
| `--out=FILE`       | Куда писать (по умолчанию — stdout)                    | stdout          |

## Два режима генерации (`--per-request`)

- **`--per-request=1` (по умолчанию) — режим стоп-маркера.** Один договор на
  запрос; генерация завершается стоп-последовательностью `<<<КОНЕЦ>>>`. Точные
  границы единиц, но N договоров = N запросов (медленнее на больших списках).
- **`--per-request=K` (K>1) — пакетный режим.** K договоров за один запрос,
  разделённые `=====`; стоп-маркер не используется, `--max-tokens` масштабируется
  на K. Число запросов падает с N до ⌈N/K⌉ — главный рычаг скорости. Модель может
  вернуть не ровно K (результат обрезается до `--count`).

## Оптимизация (модель и токены)

- **Модель + отключённые рассуждения.** По умолчанию `glm-4.5-flash` с
  `thinking: disabled` (через ядро): нет reasoning-токенов (экономия) и корректная
  работа стоп-маркера. Reasoning-модели (glm-5.1) для генерации здесь избыточны.
- **Кэширование входа.** Системный промпт стабилен — провайдер кэширует его, и
  повторные запросы почти не тратят входные токены.
- **Вход не растёт.** Запросы независимы (без накопления истории); разнообразие
  обеспечивается порядковым номером договора в промпте.
- **Параллельность.** Зафиксирована на 2: z.ai ограничивает одновременные запросы
  rate-лимитом. Повышать `--concurrency` для этого провайдера не стоит — основную
  скорость даёт `--per-request`. Сетевые сбои и 429 ядро повторяет с бэкоффом.

## Пример замкнутого цикла

```bash
# 1) сгенерировать тестовые договоры
npm run generate -- --count=20 --per-request=10 --out=/tmp/contracts.txt

# 2) извлечь из них реквизиты (в соседнем пакете)
cd ../contract-extractor
npm run extract-json -- /tmp/contracts.txt --batch=5
```
