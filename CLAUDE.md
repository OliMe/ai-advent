# CLAUDE.md

Указания для работы с репозиторием **ai-advent** — набором консольных приложений на
Node.js + TypeScript для работы с LLM через OpenAI-совместимый API (запросы нативным
`fetch`, без сторонних SDK). Несколько независимых пакетов вокруг общего ядра `core`.

## Пакеты

| Пакет | Назначение |
| ----- | ---------- |
| `core/` | Общее ядро: HTTP-клиент к OpenAI-совместимому API, конфиг из окружения, типы, ретраи (429/5xx/сеть) с бэкоффом, `disableThinking`. Плюс движок памяти, сессии, профили, задачи и автономный пайплайн. Остальные пакеты импортируют его напрямую. |
| `llm-cli/` | Консольный чат с моделью: одиночный запрос и интерактивный режим, слоистая память, мультипрофили, задачи и пайплайн (`/run`). **Основной фокус разработки.** |
| `contract-extractor/` | Извлечение реквизитов сторон из договоров (`extract-json`, `list-text`). |
| `contract-generator/` | Генерация синтетических договоров для тестовых данных. |
| `task-solver/` | Решение задачи четырьмя стилями промптинга + оценка GLM. |
| `hf-multi/` | Параллельный вызов нескольких моделей HuggingFace на один промпт. |
| `mcp-probe/` | Песочница-исследование MCP: клиент к любому серверу (stdio/Streamable HTTP, настройка через `.env`), печать списка инструментов или результата вызова инструмента. **Исключение**: использует официальный `@modelcontextprotocol/sdk`. |
| `yandex-ocr-mcp/` | MCP-**сервер** распознавания текста (Yandex Vision OCR): инструмент `recognize-text` (файл/URL/base64), конфиг из `.env`. Два транспорта: stdio (`cli.ts`) и Streamable HTTP (`http.ts`, express + bearer-auth `auth.ts`) для деплоя на VPS (Dockerfile + systemd). **Исключение**: `@modelcontextprotocol/sdk` + `zod` + `express`. Тонкие `server.ts`/`cli.ts`/`http.ts` вне покрытия. |
| `mcp-client/` | Адаптер MCP-серверов к абстракции `ToolSet` ядра: подключение (stdio/HTTP), агрегирование инструментов (неймспейс `сервер__инструмент`), рантайм add/remove. `connectionFactory(onElicit)` — клиент объявляет capability `elicitation` и переадресует серверные запросы `elicitation/create` обработчику приложения (подтверждение операций от сервера). **Исключение**: официальный `@modelcontextprotocol/sdk`; тонкий `connection.ts` вне покрытия. |
| `filesystem-mcp/` | MCP-**сервер** файловой системы с «мягкой» песочницей (allow-list). Инструменты: `read_file`/`write_file`/`append_file`/`list_dir`/`delete_path`. Разрешённые каталоги — позиц. аргументы или `FS_ALLOWED_DIRS`. Путь вне allow-list НЕ отклоняется сразу: через `confirm` (MCP **elicitation**) спрашивается подтверждение пользователя; без обработчика — отказ. Защиты delete: без рекурсии, нельзя удалить корень («стереть всё» невозможно). `classifyPath` (резолв без отказа) + ворота `authorizePath`; `write`/`append` создают родительские каталоги. **Исключение**: `@modelcontextprotocol/sdk` + `zod`; тонкие `server.ts`/`cli.ts` вне покрытия. |

| `pr-reviewer/` | Автоматическое **AI-ревью PR** (День 32). По появлению нового PR: `fetchChanges` (diff + изменённые файлы из API — точные позиции строк для комментариев) → `groundDocs` (RAG по докам проекта через пакет `rag`, мягкая деградация на сырые доки без эмбеддингов; **кэш индекса**, см. ниже) → `generateReview` (LLM → структурные находки `{file,line,severity,title,body}`+summary через общий `core.completeStructured`) → `validateFindings` → `publish` (инлайн-комментарии у строк + сводка). **Diff из API, а не `git diff`** — только API PR даёт позиции строк, принимаемые для инлайн-комментариев. **Анти-галлюцинация — целевая (`validateFindings`), не цитатный гейт Дня 24:** гейт даёт один текст, а не список находок с координатами; здесь находка идёт инлайн, ТОЛЬКО если `(файл,строка)` реально в `commentableLines` diff (добавленные+контекстные новой версии), иначе — в сводку (координаты не выдумываются). **Платформенный шов** `ReviewPlatform` (`fetchChanges`/`publish`) + `github.ts`; база API настраиваемая (`GITHUB_API_URL`/`CI_API_V4_URL`) — GitHub Enterprise и корпоративный GitLab из коробки (GitLab-адаптер — задел). HTTP `requestJson` с ретраями по образцу `core/chat-completion-client`, `fetch` инжектируется. **Кэш индекса доков** (`index-cache.ts`): дорогая часть RAG — эмбеддинг КОРПУСА доков (на CPU-эмбеддере минуты: замер на этом репо 98 чанков ~207с), а не запроса (один вектор ~1с). Доки меняются редко относительно частоты PR → индекс собирается ОДИН раз. `computeIndexCacheKey` — по СОДЕРЖИМОМУ доков (+ стратегия + модель эмбеддинга), доки сортируются по пути (порядок загрузки не влияет); правка доков инвалидирует кэш, неизменные — попадание (замер живьём: 207с→0.2с, результаты идентичны). `groundDocs.loadOrBuildIndex`: есть кэш → по ключу load, промах → build+save; эмбеддинг запроса делается всегда. `FileIndexCache` — `<dir>/<key>.json` через инжектируемый `IndexCacheIo` (реальный fs-IO — в `cli.ts`, вне покрытия; битый/отсутствующий файл → `null`=пересбор, не ошибка). **Кэш платформенно-нейтрален — лежит НИЖЕ шва `ReviewPlatform`**, ключ не содержит ничего от GitHub/GitLab, поэтому один код кэша обслуживает обе платформы; различается лишь СЛОЙ персистентности каталога в CI. Каталог — `PR_REVIEW_CACHE_DIR`, дефолт `.pr-review-cache` **ВНУТРИ рабочего дерева** (важно для GitLab: его `cache:` берёт только пути внутри проекта, в отличие от `actions/cache`, который берёт любой путь) — так одна настройка работает у обеих платформ. Персист в CI: `actions/cache` (GitHub, ключ `hashFiles` по докам) в обоих workflow; у GitLab-адаптера допишется `cache:`-строфа (кэш-код не трогается). **Фоновый пре-варм** (`cli.ts --warm-cache` + workflow `ai-review-index.yml` по `push` в доки в main): собирает индекс и выходит БЕЗ обращения к модели/API PR (нужен только эмбеддер — `loadConfig` в warm-ветке не зовётся, идёт до неё), кладёт в тот же кэш по тому же ключу → даже первый PR после правки доков попадает в тёплый кэш (ключ warm-файла == ключ, который считает ревью; ключ не зависит от запроса/topK). Прогрев **строгий** (`warmDocsIndex`, БЕЗ мягкой деградации `groundDocs`): ошибка/таймаут эмбеддера ПРОБРАСЫВАЕТСЯ → job краснеет, а не «зелёный с пустым кэшем». **Таймаут — критичен (поймано живым CI-прогоном):** эмбеддинг корпуса на CPU-эмбеддере — минуты, а дефолт `LLM_REQUEST_TIMEOUT_MS=60000` обрывает ПЕРВЫЙ батч → `groundDocs` молча деградировал, кэш оставался пуст (в логе `Path Validation Error: .pr-review-cache does not exist`). Поэтому во ВСЕХ трёх workflow задан `LLM_REQUEST_TIMEOUT_MS: '300000'`. **Пост-обработка находок** перед публикацией (`postprocess.ts`, чисто): дедуп по (файл,строка) — на строке один комментарий, самая серьёзная категория; порог `PR_REVIEW_MIN_SEVERITY` (дефолт nitpick=не режет) — мелочи в сводку; лимит `PR_REVIEW_MAX_INLINE` (дефолт 20) — инлайн сортируется по важности, сверх потолка в сводку. **Идемпотентность** (`idempotency.ts`+`github.ts`): триггер включает `synchronize` (ревью на каждый пуш в PR); свои комментарии помечены маркером `<!-- ai-review -->` (`hasAiMarker`). На каждом прогоне `publish` (самодостаточный): СНИМАЕТ свои прежние ИНЛАЙН-комментарии (GET `/pulls/{n}/comments`→маркер→DELETE) и ставит актуальные ОТДЕЛЬНЫМИ комментариями (POST `/pulls/{n}/comments` с `commit_id`=head sha, `line`/`side:RIGHT`); СВОДКУ держит ОДНИМ issue-комментарием и обновляет на месте (PATCH `/issues/comments/{id}`, лишние свои удаляет, нет — POST). **Ревью-объекты (POST reviews) НЕ используются** — submitted-ревью через API не удаляются, их сводки копились бы стопкой «reviewed» (воспроизведено на демо-PR: инлайн-дубли ушли, а сводки накапливались). Сопоставление «та же строка» тоже отвергнуто — хрупко (модель приписывает дефект к РАЗНОЙ строке ±1, GitHub пересчитывает позиции при изменении ханка). Компромисс — сброс «Resolve conversation» на инлайн бота. Чужие комментарии не трогаются. Триггер — `on: pull_request: types:[opened,reopened,synchronize]`. **Распространение** — копируемый workflow (`.github/workflows/ai-review.yml` для этого репо, `docs/ai-review-workflow.yml` для чужого: чекаутит и целевой репо, и ai-advent). В CI LLM только облачная/VPS-шлюз (localhost Ollama с раннера недоступен); PR из форков — read-only токен, инлайн-постинг только для веток самого репо. Импортирует `core`+`rag`, БЕЗ сторонних рантайм-зависимостей (в CI не нужен `npm install`). Тонкий `cli.ts`/адаптеры вне покрытия. |

| `rag/` | RAG-**индексатор** (День 21). Пайплайн: источник → чанкинг → эмбеддинги → JSON-индекс с метаданными. ## Архитектура `core`

- `chat-completion-client.ts` — `ChatCompletionClient`: `completeWithUsage` (с
  `AbortSignal`-таймаутом) и `streamWithUsage` (таймаут по простою, дельты через колбэк).
  Ретраи 429/5xx/сети, `disableThinking`.
- `http-request.ts` — `requestJson` (обобщённый HTTP к JSON-API с ретраями 429/5xx/сети и таймаутом,
  `fetch`/пауза инжектируются) + типы `FetchLike`/`HttpResponse`/`RequestOptions`. Провайдеро-независим —
  для REST-API (GitHub/GitLab/трекеры); им пользуются `pr-reviewer` и `support-mcp`, не дублируя ретраи.
- `embeddings-client.ts` — `EmbeddingsClient`: OpenAI-совместимый `/embeddings` (батч-вход,
  сортировка по `index`), провайдеро-независим (`EmbeddingsConfig` из `loadEmbeddingsConfig`,
  env `LLM_EMBEDDINGS_URL/MODEL/API_KEY`); `Authorization` только при ключе (Ollama без ключа).
  Для RAG-индексатора (`rag/`). `http-retry.ts` — общие ретраи 429/5xx/сети для обоих клиентов.
- `config.ts` (`loadConfig`/`AppConfig`, `loadEmbeddingsConfig`/`EmbeddingsConfig`), `types.ts`
  (`ChatMessage`, `GenerationLimits`, `Usage`, `ResponseFormat` и т.д.).
- `conversation.ts` — `Conversation`: переиспользуемый раннер диалога над клиентом.
  Держит транскрипт, на `ask(userInput, onDelta?)` шлёт окно истории и возвращает ответ;
  `onDelta` → стрим. На нём строятся интерактив, этапы пайплайна и будущие субагенты.
- `tokens.ts` — оценка токенов, бюджет истории, обрезка скользящим окном, стоимость.
  `formatUsageStats` печатает строку статистики под ответом; опц. `elapsedMs` добавляет
  время и скорость генерации (ток/с) — для ручного сравнения квантов/параметров (см.
  `docs/local-llm.md`). llm-cli замеряет время последнего прохода генерации в `produce`.
- Сессии: `session.ts` (`Session`, `createSession`, `sessionId`) + `session-store.ts`
  (`FileSessionStore`, атомарная запись). Ветвление/переключение сессий.
- Память (движок): `memory-strategy.ts` (короткая: window/summary/facts),
  `memory-task.ts` (`TaskMemory`; извлечение отдаёт ПОЛНЫЙ обновлённый список фактов задачи,
  `applyDetails` их замещает — НО пустой список НЕ затирает уже накопленные детали: на
  офф-топике/болтовне модель часто возвращает `task: []`, это «нет новых фактов», а не «забудь
  задачу»), `memory-profile.ts` (`ProfileMemory`),
  `memory-invariants.ts` (`InvariantsMemory` — глобальные жёсткие ограничения),
  `memory-manager.ts` (`MemoryManager` — оркестрирует инварианты+профиль+задачу+короткую,
  подмешивает блоки и директивы персонализации/инвариантов, извлекает память из реплик,
  консолидирует профиль). Извлечение попутно метит **ход-воспоминание** (`MemoryWriteReport.recall`
  — пользователь просит вспомнить/повторить уже сказанное, а не задаёт новый знаниевый вопрос);
  сигнал для grounded-режима llm-cli отвечать из истории, а не искать заново (см. День 25). Хранилища: `profile-store.ts`, `task-store.ts`,
  `invariants-store.ts` (глобальный `~/.llm-cli/invariants.json`).
- Пайплайн задач: `task-run.ts` (`TaskRun`/`Stage`/артефакты/`createRun` + **явный
  автомат жизненного цикла**), `run-store.ts` (`FileRunStore`: `<id>.json` + каталог
  файлов-артефактов), `pipeline-stages.ts` (раннеры этапов + ленивый разбор),
  `pipeline.ts` (`runPipeline` — оркестратор-автомат), `stage-team.ts` (**команда
  агентов на этап**: оркестратор подбирает роли, ограниченно-параллельный веер,
  синтез), `json.ts` (общий ленивый разбор JSON из ответов модели),
  `pipeline-schemas.ts` (JSON-схемы этапов + `structuredLimits`). Подробнее — ниже.
- Явные переходы между состояниями (`task-run.ts`): таблица `ALLOWED_STAGE_TRANSITIONS`
  (рёбра между этапами; перепрыгнуть этап нельзя) + `stagePrerequisiteMet` (есть ли
  артефакт предыдущего этапа: план — только после требований, завершение — только после
  пройденной проверки). `canTransition` сводит их в вердикт, `applyTransition` —
  единственная точка смены этапа: на недопустимом переходе **бросает**
  `InvalidTransitionError` (через неё ходит `transition()` в `pipeline.ts`). `repairStage`
  при возобновлении чинит несогласованный прогон (этап «впереди» своих артефактов —
  правленый/повреждённый файл) откатом назад, не двигая вперёд; драйвер сообщает об этом
  хуком `onStageRepair`. Так ассистент не проскочит этап ни в нормальном ходе, ни после
  паузы.

## Архитектура `llm-cli`

- `cli.ts` → `index.ts` `main()`: разбор аргументов (`args.ts`), режим одного запроса
  (`chat.ts` `runOnce`) или интерактив (`interactive.ts`).
- `interactive.ts` — цикл readline + **реестр команд** (`{matches, run}`, первая
  подходящая по `matches`; порядок важен: точные перед префиксными). Команды: `/help`,
  `/sessions`, `/branch`, `/switch`, `/reset`, `/system`, `/file`, `/temp`, `/task*`,
  `/profile*`, `/run*`, `/mcp*`. Собирает `MemoryManager`, мост к задачам и `RunController`.
- MCP-инструменты агентам (Фаза 3a): `mcp-store.ts` (`FileMcpStore` над `~/.llm-cli/mcp.json`)
  + `McpToolSet` из пакета `mcp-client` собираются в `main()` (если не `--no-mcp`) и передаются
  в `runInteractive`. Инструменты идут **планировщику и исполнителю** (через `ConversationFactory`
  → `StageContext.tools`); вызовы печатаются (`onToolCall`). Команды `/mcp`, `/mcp add <имя>
  <команда|URL>`, `/mcp remove <имя>`, `/mcp reload` — живое подключение/отключение + запись в
  `mcp.json`; серверы подключаются на старте, закрываются на выходе (сбой сервера не валит app).
  Tool-use ядра — в `core` (`ToolSet`, агентный цикл `Conversation`), провайдеро-независимо.
  Чат тоже умеет инструменты (Фаза 3b): при подключённых серверах ход чата идёт агентным
  циклом `completeWithTools` (`chat.ts`) — БЕЗ стрима (tool_calls в потоке сложны), обёрнут в
  контролёр инвариантов; расход токенов суммируется по раундам. Требуется провайдер с
  function-calling (проверено на GLM и DeepSeek). Длина агентного цикла — `AppConfig.maxToolRounds`
  (env `LLM_MAX_TOOL_ROUNDS`, дефолт 20): хватает на длинный кросс-серверный флоу (геолокация →
  места → файл → планировщик). После хода печатается **трасса вызовов** (`formatToolTrace`:
  «🧭 a → b → c») — видно выбор инструментов и порядок маршрутизации по серверам.
- RAG-директива (`rag-directive.ts`, День 22): если среди инструментов есть `…search_docs`
  (подключён `rag-mcp`), в ведущие system-сообщения добавляется `ragSearchDirective` — агент
  вызывает поиск по документам на вопросы о репо/сайте/папке (с source из вопроса) и отвечает
  строго по фрагментам. **День 24**: директива требует ОБЯЗАТЕЛЬНЫЙ формат из трёх секций
  (Ответ/Источники `source › section · chunk_id`/Цитаты — дословные выдержки) и режим «не знаю» при
  пустом/слабом контексте (`⚠ Низкая уверенность`); `compact`-режим (`RAG_ANSWER_COMPACT=1`) —
  короткие источники/одна цитата для длинного чата. `rag-answer.ts` `parseSearchResult` толерантно
  разбирает вывод `search_docs` в чанки (`chunk_id`/текст) + уверенность. **Гейт цитат
  (`citation-guard.ts`, День 24 Этап 3):** после RAG-хода `resolveRagAnswer` — при пустом контексте
  или слабой уверенности (все результаты `lowConfidence`) отдаёт «не знаю» (`RAG_DONT_KNOW`), иначе
  ЛОКАЛЬНО (без LLM) проверяет ответ: источники ⊂ найденных (все — подделка источника недопустима) И
  **≥1 дословная цитата-якорь** (нормализованная подстрока чанка, ≤500 симв.); провал → перегенерация
  (до 5 раз) с замечанием, не сошлось → безопасный фолбэк `RAG_UNVERIFIED`. Строковая сверка
  неподделываема (в отличие от LLM-судьи). **День 25 п.2 (синтез):** правило смягчено — весь ответ
  дословным НЕ требуем (синтез/пересборка процедуры в теле «Ответ» — норма), нужен лишь ОДИН дословный
  якорь; лишние не-дословные записи в «Цитаты» (собранные команды) терпим. Раньше «ВСЕ цитаты
  дословные» ложно валило синтез («расскажи порядок действий» → «не знаю»).
  **Адресный фидбэк (`validateCitations` собирает ВСЕ провалы за проход, не первый; `citationFeedback`):**
  замечание перегенерации называет КАЖДЫЙ непринятый источник/цитату + даёт жёсткий шаблон трёх секций
  (частый провал — модель вовсе опускает Источники/Цитаты; явный образец надёжнее общей просьбы). Так
  модель правит все провалы разом, а не по одному за перегенерацию — меньше ложных фолбэков.
  `normalizeForMatch` оставляет только буквы/цифры (пунктуация → пробел): «дословность» = «те же
  слова подряд», устойчиво к тому, что модель «облагораживает» кавычки/тире/точки. `parseAnswerSections`
  понимает и `Цитаты:`, и markdown `## Цитаты` без двоеточия. Два слоя «не знаю»: порог уверенности
  ловит офф-топик на БОЛЬШОМ корпусе (лучший косинус проседает), цитатный гейт — на маленьком (нечего
  дословно процитировать по теме). Эти нюансы (кавычки/тире, markdown-заголовки, лимит цитаты,
  калибровка `RAG_CONFIDENCE_MIN=0.6`) найдены эвалом Дня 24 на Оркестре (10/10) и bumblebee (10/10). **Опциональный гейт достоверности (`faithfulness.ts`, Этап 4, тумблер
  `RAG_FAITHFULNESS_CHECK=1`):** поверх локального — LLM-судья (`FAITHFULNESS_CHECKER_SYSTEM`) сверяет,
  что утверждения ответа опираются на фрагменты; провал → перегенерация, не сошлось → фолбэк. По
  умолчанию выкл (семантическая проверка капризна — тумблер для сравнения «с ним и без»). **День 25
  п.2:** промпт судьи смягчён — принимает обоснованный СИНТЕЗ/пересборку из нескольких фрагментов
  (процедура, порядок действий, сводка; дословность не требуется), бракует ТОЛЬКО противоречащее или
  невыводимое (выдумку). Гейт только в интерактиве (в `runOnce` инструментов/RAG нет). **Grounded-режим мини-чата (`grounded.ts`, День
  25):** `/rag <src…>` привязывает источники к сессии (`session.ragSources`, персистится). Тогда на
  каждой **содержательной** реплике (`isConversationalReply` отсеивает привет/спасибо/да-нет) llm-cli
  ДЕТЕРМИНИРОВАННО зовёт `search_docs` по КАЖДОМУ источнику (`forcedRagSearch`), запрос обогащён ТОЛЬКО
  целью задачи (`groundedFocus`/`buildGroundedQuery`; инварианты в запрос НЕ идут — это правила
  поведения, а не поисковый контекст, они лишь зашумляли эмбеддинг), фрагменты кладутся в контекст, дальше
  ответ **TOOL-FREE** (`askModel`, не `completeWithTools`) на **НИЗКОЙ температуре** (`RAG_ANSWER_TEMPERATURE`,
  дефолт 0.2 — `resolveRagAnswerTemperature`; та же и у перегенерации гейта) + гейт Дня 24: синтез строго
  по фрагментам, творчество вредит (точность/достоверность + меньше шума на повторах); не связано с
  `response_format` → безопасно для всех провайдеров. Так «при каждом вопросе ищет
  в базе» — надёжно, а не по усмотрению модели. **Grounded-ответ tool-free (важно для слабых моделей):**
  фрагменты уже добыты форс-поиском, модели остаётся лишь синтезировать ответ по ним — полный агентный
  цикл давал слабой локальной модели (qwen 7B) «играться» с посторонними инструментами (scheduler и
  т.п.) вплоть до ВРЕДНЫХ побочных действий из вопроса про доки; первый проход теперь консистентен с
  перегенерацией гейта (она и так tool-free). Не-grounded (в т.ч. «источник назван в запросе») — обычный
  агентный цикл. **Предохранитель grounded (`RAG_SEARCH_UNAVAILABLE`):** если форс-поиск вернул ПУСТО
  (`forcedResults.length === 0` — инструмент `search_docs` пропал из `specs()`, напр. rag-сервер stdio
  отвалился в рантайме), grounded НЕ пускает сырой ответ модели (иначе слабая модель галлюцинирует
  несуществующие профили/источники — воспроизводилось на qwen 7B) — короткозамыкает безопасным
  сообщением «не удалось обратиться к базе, `/mcp reload`». Так grounded НИКОГДА не выдаёт необоснованный
  ответ, а сбой сервера становится виден вместо правдоподобной лжи. **Ход-воспоминание (День 25 Этап 3, классификация вопросов):** «напомни/что мы
  решили/с чего начали…» не гонится через форс-поиск и цитатный гейт — сперва проба ВОСПРОИЗВЕСТИ
  прошлый ответ из истории (`isRecallTurn` = гибрид: LLM-флаг `MemoryWriteReport.recall` ⋁ лексические
  `RECALL_MARKERS`; вызов с `RECALL_SYSTEM_PROMPT`, temp=0 — дословно, с секцией «Источники», без
  поиска/гейта). Нет ответа в истории → модель возвращает `RECALL_SENTINEL` (`isRecallFallback`), и
  клиент МОЛЧА откатывается на обычный grounded-путь (сентинел пользователю не показывается). Так
  вопрос-воспоминание отвечается из контекста (а не ложным «не могу подтвердить»), а неизвестное —
  свежим поиском. Только на первой генерации хода (не при перегенерации контролёром инвариантов).
  **Валидация recall якорем (`recallAnchoredInHistory`):** LLM-флаг `recall` ЛОЖНО-ПОЛОЖИТЕЛЕН на
  слабых моделях (замер: qwen2.5:7b метит обычный вопрос «какие профили…?» как recall 5/5) — при
  включённой памяти это уводило grounded-вопрос в recall-ветку, где слабая модель ГАЛЛЮЦИНИРУЕТ
  вместо честного сентинела (нет форс-поиска, нет `🔎`, фейковый источник). Поэтому recall-ответ
  показывается ТОЛЬКО если заякорен — содержит непрерывную нормализованную подстроку (≥6 слов,
  `normalizeForMatch` из citation-guard) из ПРЕЖНИХ ассистентских сообщений. Нет якоря (в т.ч.
  первый ход — вспоминать нечего) → откат на grounded. Детерминированно, НЕ зависит от честности
  модели (сентинелу доверять нельзя). Гибкость сохранена: проба recall идёт всегда (LLM-флаг не
  трогаем), отсекается лишь неподтверждённый результат; настоящий recall на сильной модели проходит
  (он и правда дословно повторяет историю). Асимметрия рисков: ложный откат→grounded безвреден,
  ложный recall→галлюцинация вреден — потому при сомнении выбираем grounded.
  `RAG_QUIET=1` прячет логи RAG-поиска (🔎/🔧) — в grounded на ход несколько
  поисков, шумно; ответ с источниками остаётся. Нет инструмента — директивы нет. Результат
  `search_docs` печатается в чат **сводкой** (`formatRagResultForDisplay` в `reportToolResult`):
  строка-трасса стадий (🔎) + заголовки источников `[n] source › file › section (score)`, без тел
  фрагментов — чтобы разница режимов (rerank/rewrite/порог) была видна пользователю, а не только
  модели (иначе сильная модель сглаживает её в финальном ответе).
- Подтверждения от серверов (MCP elicitation): `elicitation.ts` — `ElicitationBridge`
  связывает обработчик (создаётся рано, в `main()`, и идёт в `connectionFactory` →
  `McpToolSet`) с запросом к пользователю через readline (`readlineConfirm`), подставляемым в
  `runInteractive`, когда готов ввод. Пока запрос не подставлен — безопасный `decline`. Так
  `filesystem-mcp` может в рантайме спросить разрешение на операцию вне песочницы (⚠ да/нет).
- `chat.ts` — `askModel`/`streamAnswer` (+ спиннер «думает…»), `runOnce`.
- `run-flow.ts` — `RunController` (драйвер `/run`): фабрика диалогов агентов,
  сбор требований, прогон пайплайна с печатью этапов и подтверждением, пауза по Ctrl+C.
- `run-task-bridge.ts` — `MemoryRunBridge` (поверх `MemoryManager` через узкий фасад
  `TaskMemoryFacade`, DIP): связывает прогон с задачей сессии.
- `formatters.ts` (тексты/справка/рендер этапов), `replies.ts` (да/нет/список),
  `paths.ts` (каталоги `sessions`/`tasks`/`profiles`/`runs`), `files.ts`, `errors.ts`,
  `session-flow.ts`.
- `index.ts` — barrel: реэкспортит `core` + локальные модули (тесты импортируют отсюда).

## Модель «проект» (ассистент разработчика, День 31)

Проект — **сущность ядра** (`core/project-context.ts`), а не настройка llm-cli: на неё опираются и
вопросы о проекте, и агенты пайплайна. Цель — понимать ЛЮБОЙ указанный проект (не только этот репо)
и **несколько сразу**: фича живёт в нескольких репозиториях (бэк + фронт + инфра).

- `ProjectContext { root, name, origin?, docSources[], packageManager?, commands{test,build,lint,start} }`.
  Всё выводится из репозитория: `detectProjectRoot` (вверх до `.git`), `discoverDocSources` (README*/
  CLAUDE.md/AGENTS.md, каталог `docs`, `openapi*`/`schema.*`/`*.proto`), `detectProjectCommands`
  (`scripts` из `package.json` — пайплайну нужно знать, ЧЕМ проверять), `detectOrigin` (из
  `.git/config`, без запуска git — сущность не зависит от внешнего процесса). Логика с инъекцией
  `ProjectIo` (реальный — `nodeProjectIo`). `formatProjectContext`/`formatWorkspace` — карточки для
  промптов агентов.
- **Сессия хранит `projects?: string[]`** (корни; персистится). Команды `llm-cli`: `/project`
  (список + живая ветка), `/project add <путь|git URL>`, `/project remove <имя|путь>`, `/project off`.
  Ничего не привязано → **автодетект по cwd** (`resolveWorkspace` в `project.ts`), о чём сообщается
  строкой `📁 Проект …` — «понимание проекта» не должно быть молчаливым допущением.
- **Разделение труда: доки → RAG, код → инструменты.** В индекс идёт только документация
  (`docSources`); код добывается точечно через `git-mcp` (`git_grep`/`read_file`/`git_list_files`).
  Причина: код большой, быстро устаревает в индексе и отлично ищется точным поиском, а документация
  выигрывает от поиска по смыслу. Так ассистент работает с любым репо без долгой предындексации кода.
- **Привязка проекта = разрешение на чтение** (`git-allowlist.ts`): `/project add` дописывает корень
  в `GIT_ALLOWED_REPOS` git-сервера (`mcp.json`) и переподключает его (`removeServer`+`addServer`).
  Иначе мягкая песочница спрашивала бы подтверждение на КАЖДЫЙ вызов инструмента (за один `/ask` их
  5–8) — так фича была неюзабельной. Пишем в `env`, а НЕ в позиционные аргументы: там лежит путь к
  `cli.ts` и ручные настройки пользователя, и клиент не должен их ни трогать, ни пытаться отличать
  от своих (найдено живым смоуком: дописав первый проект в args, я лишил сервер дефолтного cwd).
  Сервер (`git-mcp/config.ts`) поэтому ОБЪЕДИНЯЕТ источники allow-list: аргументы + `GIT_ALLOWED_REPOS`
  + рабочий каталог + кэш клонов, причём аргументы и cwd идут ПЕРВЫМИ (первый элемент — репозиторий
  по умолчанию: иначе привязка чужого клона молча делала бы дефолтом его). Репозиторий вне списка
  по-прежнему требует подтверждения. В обычном чате карточки проектов (`formatWorkspace`) идут в
  ведущие system-сообщения — без корней модель подставляет в git-инструменты ИМЯ проекта вместо пути,
  и получает «это не git-репозиторий». `/project remove` и `/project off` СИММЕТРИЧНО снимают
  разрешение (`revokeRepositoryInGitServer`): убирают корень из `GIT_ALLOWED_REPOS` и переподключают
  сервер, возвращая конфиг в исходное состояние (опустевший список → ключ env удаляется целиком).
  Трогается только `env`-часть клиента: репозиторий из ручных аргументов или рабочий каталог не в
  env (`absent`) — его не убираем, это не наша запись.
- **URL-проект сводится к локальному** (`project-source.ts`): клон в кэш `~/.llm-cli/projects/<хэш>/<имя>`
  — хэш ОТДЕЛЬНЫМ каталогом, потому что имя проекта = `basename(root)`, и «is-odd» должен остаться
  «is-odd» (этим именем пользователь адресует проект), а хэш лишь разводит форки. Клон
  `--filter=blob:none`, а НЕ `--depth=1`: неполная история сделала бы `git log`/`git diff`
  бесполезными — а ими ассистент и понимает проект. Повторная привязка → `git fetch`, не новый клон.
  `realGitRunner` ставит **`GIT_TERMINAL_PROMPT=0`**: иначе git на приватном/несуществующем репо ЖДЁТ
  логин — в интерактивном CLI это выглядит как зависание (найдено живым смоуком).

### `/ask <вопрос>` — ответы о проекте (День 31)

`/help` — по-прежнему справка; вопрос ассистенту разработчика — `/ask <вопрос>`. Разбирается ДО
реестра команд (`parseProjectQuestion`), чтобы ход переиспользовал общий путь реплики (память,
инварианты, статистика), а не дублировал его внутри команды.

**Ход полностью ДЕТЕРМИНИРОВАННЫЙ — инструменты зовёт `llm-cli`, а не модель** (`project-answer.ts`):
карточки проектов + git-контекст (`collectGitContext`: ветка+статус) → форс-поиск по докам всех
проектов (`forcedRagSearch`) → модель называет 1–3 шаблона для grep (`CODE_SEARCH_SYSTEM`,
`parseCodePatterns`, temp 0) → форс-`git_grep` (`forcedCodeSearch`) → чтение файлов
(`forcedFileReads`) → **tool-free синтез** (`askModel`, низкая температура) → цитатный гейт Дня 24.

Найдено живыми прогонами (qwen2.5:7b), каждое решение — цена ошибки:
- **Агентный цикл здесь ВРЕДЕН.** Отданные модели инструменты → она десятками раундов читала
  ВЫДУМАННЫЙ `gates/citation_gate.txt`, пока не кончился лимит раундов; ответа не было вовсе. Тот же
  вывод, что в grounded Дня 25: слабой модели даём синтез, а добычу фактов делаем сами.
- **Шаблоны grep — по РЕАЛЬНОМУ списку файлов** (`projectFileListings` → в промпт). Без списка модель
  транслитерировала русские слова (`gate_citata`) и grep не находил ничего.
- **Шаблон-имя файла разрешается в путь** (`filesMatchingPatterns`) и файл читается ЦЕЛИКОМ. Иначе
  grep по `citation-guard.ts` давал лишь УПОМИНАНИЯ (импорты, доки), а сам файл-определение не
  читался — и отвечать было нечем.
- Файлы из совпадений grep (`filesFromHits`, до 3, чаще встречающиеся раньше) — по одной строке
  совпадения код не понять.

**Доказательная база хода (`evidence.ts`):** результаты `read_file`/`git_grep`/`git_list_files`/
`git_diff`/`git_log` превращаются в чанки (`chunk_id` = `проект › путь`) и передаются в
`resolveRagAnswer` как `extraChunks` — цитатный гейт (источники ⊂ найденного + дословный якорь)
работает и для ответов ПО КОДУ. Код в RAG-индекс не кладётся, поэтому без этого гейт валил бы любой
ответ о коде в `RAG_UNVERIFIED`. Сверка строковая → выдумать код нельзя (проверено живьём: модель
сослалась на документацию git — гейт завернул). «Не знаю» — только когда доказательств НЕТ вовсе
(слабый док-контекст не повод молчать, если прочитан код).

Отказы вместо выдумки: нет проекта → `ASK_NO_PROJECT`, нет MCP → `ASK_NO_TOOLS`.

## Модель «задача» (память + пайплайн)

Задача — **единая сущность**, а `/task` и `/run` — два действия над ней: «вести» и
«исполнить».

- `/task [текст]`, `/tasks`, `/task switch|done|delete` — ведение задачи: по ходу диалога
  она копит факты (детали) и подмешивается в контекст чата. Директива персонализации
  велит модели при новой/неоднозначной задаче **сначала задать уточняющие вопросы**, и
  только потом решать. `/task <текст>` **только устанавливает** текущую задачу (цель в
  память) — пайплайн НЕ запускается автоматически; исполнение — отдельной командой `/run`.
  (Авто-предложенная задача из реплики по подтверждению «да» — запускается сразу, как раньше.)
  **Авто-предложение задачи подавляется при АКТИВНОЙ незавершённой задаче** (`applyExtraction`:
  условие `this.tasks.current()?.status !== 'active'`): `isNewTask` — ненадёжный LLM-флаг (слабая
  модель метит им обычный знаниевый вопрос по ходу задачи → «Похоже на новую задачу?» на каждый
  вопрос). Пока задача идёт, смену пользователь делает явно (/task); нет задачи или она done —
  предложение работает. Асимметрия рисков: ложное предложение мешает, пропуск безвреден (есть /task).
- `/run [id|описание]` — исполнить задачу пайплайном: без аргумента — текущую, по id —
  существующую, по описанию — новую. Прогон связан с задачей через `taskId`.

Поток `/run`:

1. **Сбор требований** (до планирования): агент-аналитик задаёт уточняющие вопросы по
   одному; ответы пишутся в детали задачи и в транскрипт. Нет вопросов — пропускается.
2. **Фиксированный пайплайн** `planning → execution → verification → completion`
   (пропуск запрещён, каждый этап даёт артефакт). Память задачи (детали + профиль)
   подаётся в планирование и выполнение через `memoryContext`.
3. Провал проверки или отказ на завершении → авто-возврат в `execution` до `maxRetries`
   (по умолчанию 2); исчерпано → пауза. **Завершение требует подтверждения пользователя.**
4. По подтверждению итог дописывается в детали задачи, задача помечается `done`.

Управление прогоном: `/runs`, `/run status [id]`, `/run continue [id]`, `/run edit
<правка>`, `/run abort`. **Ctrl+C** во время прогона ставит его на паузу на границе
этапа (не выходит из приложения). Полный читаемый результат каждого этапа печатается в
консоль и пишется в транскрипт сессии (видно в истории, идёт в контекст чата).
`--ephemeral` держит сессии/задачи/прогоны в памяти, без записи на диск.

Структурированный вывод этапов (opt-in, `pipeline-schemas.ts`): `LLM_STRUCTURED_OUTPUTS=1` →
`AppConfig.structuredOutputs` → `PipelineDeps`/`StageContext.structuredOutputs` → `structuredLimits`
кладёт `response_format: json_schema` в `limits` диалогов планирования (соло + синтезатор), проверки
и завершения. **Дефолт — выключено**, и это ЖЁСТКИЙ инвариант: `response_format` ломает z.ai/GLM
(вырезает литерал «json»), поэтому провайдера НЕ детектим по `base_url` — тумблер поднимает
пользователь под свою модель (Ollama умеет; проверено живой пробой). Выключено → `limits` остаётся
`undefined`, запрос байт-в-байт прежний. Толерантный парсер (`extractJsonObject`) — фолбэк на обоих
путях. Выполнение схемой НЕ ограничиваем (плоский текст: код ломает экранирование JSON-строки);
вместо этого `parseExecution` делает **детерминированный ремонт обёртки** (`{name, implementation,
jsdocComments}` → код) — правило намеренно узкое: разворачиваем, только если КАЖДЫЙ ключ объекта из
белого списка (поле-результат или поле-описание), иначе задача, чей результат сам по себе JSON,
была бы искалечена. **Цена схемы:** под constrained decoding модель перестаёт вызывать
function-calling — планировщик остаётся без MCP-инструментов (замерено на Ollama).

Артефакты — структурные поля + `text`. Планирование/проверка/завершение возвращают JSON
(по умолчанию просим **в промпте, без `response_format`** — иначе z.ai/GLM вырезает литерал «json»);
парсер устойчив: берёт объект целиком или первый сбалансированный блок `{…}` из прозы
(`extractJsonObject`), фолбэк — текст/буллеты. **Выполнение возвращает плоский текст** (не
JSON: крупный код ломал экранирование JSON-строки) — `parseExecution` снимает markdown-
ограждение, `summary` = первая строка, `text` = результат; сохраняется файлом (`files[]`).
Планировщик обязан заполнять `steps` (не только `text`); при пустых шагах выполнение и
вывод берут план из `text`. **Few-shot + ремонт пункта плана (Этап 1):** промпты
планировщика/синтезатора несут пример строкового массива `steps`/`criteria`, а `coercePlanItem`
детерминированно разворачивает пункт-ОБЪЕКТ (`{name, description, …}`) в строку по белому списку
полей — слабая модель (llama3.2:3b) кладёт в `steps` объекты, и без ремонта `asStrings` молча
отдавал бы пустой план (замерено: `steps 0/criteria 0` → `8/9`). Аналитик требований тоже получил
few-shot: `suggestion` — ОДИН готовый ответ, не список «например, A/B/C» (на 3b подсказки
вырождались в перечисление вариантов). Планировщик добирает пустые критерии повторным запросом
(до 2 раз). Проверяющий сверяет результат по критериям, иначе по шагам, иначе по тексту
плана (без авто-провала на пустых критериях — иначе зацикливается на execution); совсем
пустой план → провал без вызова модели. При возврате к сбору требований устаревшие
замечания проверки очищаются.
Температуры этапов — низкие (структурированный вывод, творчество вредит формату; провайдеро-
безопасно, с `response_format` не связано): планирование/синтез плана `0.3` (немного широты для
полноты), выполнение/завершение `0.2`, проверка `0` (стабильный вердикт), аналитик требований `0.3`
(конкретные вопросы-подсказки, не мета-описания). Роли команды берут свою температуру от оркестратора
(там разнообразие уместно). Именованные константы в `pipeline-stages.ts`/`run-flow.ts`. Детали задачи в
контексте агентов ограничены бюджетом токенов (свежие важнее), потолок генерации
этапов — опц. `LLM_STAGE_MAX_TOKENS`.

Роутинг модели по роли (Этап 2, `LLM_EXECUTOR_MODEL`): роль ВЫПОЛНЕНИЯ может идти на отдельную
модель (напр. code-модель), остальные — на `LLM_MODEL`. Механизм: `CompleteOptions.model` (override
в теле запроса, `options.model ?? config.model`) → `ConversationConfig.model` → 5-й опц. параметр
`makeConversation`/`ConversationFactory` → `runExecution` берёт из `StageContext.executorModel`
(проброс `AppConfig.executorModel` → `PipelineDeps`/`RunControllerDeps` → `StageContext`). Не задан —
override `undefined`, запрос идёт на модель клиента, регресса нет (доказано на проводе). Замер (Этап 2):
`qwen2.5-coder:7b` даёт лучший код (4/4 vs 3/4 tsc-чисто на дженериках), но как ЕДИНАЯ модель ПЛОХ на
не-код ролях — планировщик выдумывает невыполнимый критерий («без `.length`/циклов»), проверяющий его
честно бракует → вечный цикл; аналитик даёт пустые/списочные подсказки. Отсюда роутинг, а не замена.
Цена на локали: смена модели за роль = переключение в Ollama (~4с на этой машине: 3 ГБ свободно, две 7b
не резидентны одновременно; при `OLLAMA_MAX_LOADED_MODELS≥2` и достатке памяти — почти бесплатно).

Проект в пайплайне (День 31, инкремент 4): два канала разной цены.
- **Карточки проектов — ВСЕМ этапам** (дёшево, всегда): `PipelineDeps.projectContext` →
  `StageContext.projectContext` → префикс промпта аналитика/планировщика/ролей/исполнителя/
  проверяющего/завершающего. План и результат должны быть про КОНКРЕТНЫЙ репозиторий (корень, ветка,
  документация, команды тестов/сборки), а не «в воздухе».
- **Документация — АДРЕСНО, где решает** (`StageContext.retrieveProjectDocs`, шов над `forcedRagSearch`
  — `core` не знает про MCP/RAG): **планирование** (план в терминах реальной структуры и соглашений) и
  **проверка** (сверка результата с соглашениями проекта). На выполнении и завершении добор данных
  ничего не решает, а стоит запросов — там только карточка. Поиск идёт по КАЖДОМУ источнику
  документации КАЖДОГО проекта (запрос = цель задачи).
- Не задано (проект не привязан / нет rag-сервера) → `undefined` → этапы работают ровно как прежде,
  регресса нет. Инструменты git уже доступны этапам через `StageContext.tools`.

Файлы и команды проекта в пайплайне (День 34): `/run` **РЕАЛЬНО правит файлы проекта и проверяет их
запуском команд**, а не только генерит текст. Активируется при привязанном проекте (нет проекта →
пайплайн как прежде, регресса нет). Механизм ОБЩИЙ («цель → агент сам работает с файлами → diff →
команды → применение»), сценарии (обновить доки/создать ADR/обновить зависимости/проверить правила) —
разные цели через него. Ключевые решения (все опциональные поля `StageContext`/`PipelineDeps`/
`RunControllerDeps` → регресс байт-в-байт):
- **Изолированная копия (git worktree).** Драйвер (`RunController.ensureWorkspace`) заводит на прогон
  рабочую копию проекта от HEAD во временном каталоге (`llm-cli/run-workspace.ts` `createRunWorkspace`):
  execution пишет в НЕЁ, verification гоняет команды ТАМ — рабочее дерево пользователя не трогается до
  подтверждения. Копия переживает паузу/`/run continue` в сессии (сверка по id прогона); завершение/
  отмена/сбой → удаляется (`dispose`, best-effort). Первичный проект — ПЕРВЫЙ из `projects()`
  (мультипроект — на потом). worktree от HEAD не видит незакоммиченных правок (агент работает от
  закоммиченного). node_modules пробрасывается симлинком (команды видят зависимости).
- **execution агентно правит файлы** (`StageContext.fileWorkspace` → `runFileExecution`): исполнителю
  даётся `FileWorkspace.tools` (`WorkspaceFileToolSet`: `read_file`/`write_file`/`list_dir`/`grep` по
  путям ОТНОСИТЕЛЬНО проекта, резолв внутрь копии, путь наружу отвергается; worktree скрыт от модели —
  деталь безопасности) вместо `ctx.tools`. Function-calling цикл `Conversation` сам решает, что читать/
  менять — цель на уровне плана (`EXECUTOR_FILE_SYSTEM`: «вноси реальные правки инструментами, а не
  описывай»). Артефакт: `text` = резюме агента + `git diff` копии, `files` = изменённые пути. Нет
  `fileWorkspace` → прежнее текстовое выполнение.
- **verification прогоняет команды проекта** (`StageContext.commandCheck` → `runProjectCommands`): гонит
  обнаруженные `test`/`build`/`lint` (`ProjectContext.commands`, `start` исключён — не завершается) на
  изменениях. Любой ненулевой код/таймаут → `passed=false` + вывод в `issues`, БЕЗ вызова модели
  (возврат в execution, агент правит в той же копии); все пройдены → их результат в контекст судьи
  (зелёные тесты — весомый довод). Команды — ТОЛЬКО обнаруженные (allow-list по построению; произвольную
  строку модель запустить не может), поэтому shell в `nodeCommandRunner` допустим; таймаут обязателен.
- **Применение — ПОСЛЕ подтверждения** (`RunController.drive`): на `completed` (значит confirmCompletion
  одобрен, пользователь видел diff на печати execution) `workspace.apply()` переносит правки в реальный
  проект КОПИРОВАНИЕМ изменённых файлов (A/M — копия, D — удаление; надёжнее `git apply`, не ломается на
  состоянии рабочего дерева), затем копия удаляется. `git diff` — только для показа. Отказ/пауза → копия
  цела для доработки.
- Швы в `core` (ядро НЕ порождает процессов/worktree): `command-runner.ts` (`ProjectCommandRunner`+
  `CommandResult`), `run-workspace.ts` (`FileWorkspace`/`CommandCheck` — только интерфейсы). Реализация —
  `llm-cli/run-workspace.ts` (логика, 100% с фейками) + тонкий `run-workspace-io.ts` (node:fs/
  child_process, вне покрытия). Проверено живыми прогонами на qwen2.5:7b: обновление README под код +
  создание нового файла ADR — правки в копии, `npm test` зелёный, diff, применение по «да».

Инварианты в пайплайне: блок инвариантов подмешивается в контекст ВСЕХ агентов этапов
(аналитик/планировщик/исполнитель/проверяющий/завершающий) — чтобы они не предлагали и не
порождали нарушающее изначально. Перегенерация при нарушении сохраняет формат/полноту ответа.

Инварианты (жёсткий контроль): вывод РЕШАЮЩИХ агентов (чат + этапы planning/execution/
completion, не аналитик/проверяющий) проходит через `enforceInvariants` (`invariant-guard.ts`)
— агент-контролёр (`INVARIANT_CHECKER_SYSTEM`) сверяет результат с инвариантами; при
нарушении называет его и агент перегенерирует (до 2 раз), иначе `InvariantViolationError`
→ в чате отказ, в `/run` пауза этапа. Пустой список инвариантов → контролёр не вызывается.

Команда агентов на этап (`stage-team.ts`): многоагентность спрятана ВНУТРЬ раннера этапа,
автомат и downstream-артефакты не меняются. Поток планирования: оркестратор
(`ORCHESTRATOR_SYSTEM`, концептуально «основной чат») по задаче+памяти решает состав
команды (`orchestrateTeam` → `parseTeamPlan`, зажат по `LLM_MAX_STAGE_AGENTS`, дефолт 4;
`≤1` — многоагентность выключена, оркестратор не зовётся). Простая задача → одна роль →
исходный одиночный `planSolo`. Сложная → роль-эксперты (архитектор/безопасность/QA/…)
работают ограниченно-параллельно (`runRoleExperts`/`mapWithConcurrency`, лимит
`LLM_STAGE_AGENT_CONCURRENCY`, дефолт 2; сбой одной роли пропускается, все упали → откат к
`planSolo`), затем синтезатор сводит вклады в ОДИН `PlanningArtifact` (поле
`contributions`, вклады пишутся файлами `planning-team-*`). Жёсткий `enforce` — на
ИТОГОВОМ синтезированном результате; инварианты идут в контекст всех ролей мягко (через
`memoryContext`). Решение оркестратора печатается хуком `onTeam` (`formatTeam`) только при
команде >1. Ленивый разбор JSON вынесен в `json.ts` (`extractJsonObject`/`parseJsonObject`).

ВАЖНО (качество плана + сходимость). Баланс: слишком строго → проверка не проходит и прогон
зацикливается; слишком мягко → вырожденный план (одна фраза, пустые steps/criteria) проходит
проверку даром. Решения:
- Оркестратор собирает команду на МНОГОГРАННой задаче (несколько аспектов/подсистем дают
  более полный результат), одну роль — для узкой задачи с единственным небольшим результатом.
- Роль-эксперты дают немного критичного, синтезатор/планировщик пишут ДЕТАЛЬНЫЙ план и
  ДИСТИЛЛИРУЮТ критерии в немного (3–6) соразмерных (не объединяя всё подряд).
- Критерии приёмки — главный гейт проверки. Если их нет (модель отдала план прозой), они
  добираются НАПРАВЛЕННЫМ запросом (узкий «выдели 3–6 критериев по плану» модель выполняет
  надёжнее, чем повторную выдачу всего плана JSON-ом). Шаги в `text` допустимы — их разворачивает
  выполнение; форсировать массив `steps` не пытаемся (GLM упирается, добор холостой).
- Проверяющий сверяет ПО СУЩЕСТВУ только заявленные критерии (без выдумывания новых и
  придирок сверх них), мелкие замечания не валят вердикт.
- Наблюдаемость: при команде печатается её состав (`🧩`) и ВКЛАД КАЖДОЙ роли в результате
  планирования (видно, что агенты отработали).
Проверено живыми прогонами задач разной сложности (тривиальная/средняя/широкая): все дают
осмысленные критерии и завершаются автономно.

Не реализовано (на потом): многоагентность на этапе ВЫПОЛНЕНИЯ (механизм `stage-team.ts`
переиспользуем, но пока подключён только к планированию); многопроектная рабочая копия Дня 34
(worktree берётся для ПЕРВОГО привязанного проекта); переживание рабочей копии через рестарт процесса
(после рестарта пересоздаётся от HEAD — правки прошлой сессии теряются).

## Частые грабли

- **`LLM_BASE_URL` — это ПОЛНЫЙ URL, а не база**: `ChatCompletionClient` бьёт `fetch(config.baseUrl)`
  как есть, поэтому значение ОБЯЗАНО быть `…/chat/completions` целиком (не `…/v1`). Эмбеддинги —
  аналогично полный `…/embeddings` в `LLM_EMBEDDINGS_URL` (поймано живым смоуком support-bot: `…/api/v1`
  давал 404).
- `verbatimModuleSyntax: true` → импорты только типов ОБЯЗАНЫ быть `import type`.
- Нет `noUnusedLocals` → неиспользуемые импорты НЕ роняют сборку (но убирать).
- Покрытие ветвей: и `summary` присутствует/пуст, и обе стороны `?:`/`??`/`||` должны
  быть пройдены тестом.
- **systemd не понимает комментарии в конце строки**: `CPUQuota=300%   # ...` — невалидное
  значение, директива тихо игнорируется (а соседняя без комментария применяется). Комментарий
  — только отдельной строкой.
- **Ollama грузит веса с `mmap = false`** и репакует их под AVX2 (`CPU_REPACK`, +~3 ГБ). Веса —
  анонимная память, не вытесняемый page cache. Поэтому 7B (4.4 ГБ) под `MemoryMax=4.6G` НЕ
  грузится вовсе (таймаут), а не «работает медленно». Считать память по `ollama ps`, а не по
  размеру файла.
- **Caddy буферизует ответ**: для SSE обязателен `flush_interval -1` в блоке `reverse_proxy`,
  иначе стрим приезжает одним куском в конце. В сборке Caddy из пакетов **нет `rate_limit`** —
  ограничитель делаем сами (`llm-gateway/rate-limit.ts`).
- **Rate limit и опрос метрик**: эндпоинт, который UI дёргает по таймеру, не должен тратить
  ведро токенов, иначе открытая вкладка сама себя банит.
- **Один сэмпл — не замер**: `llama3.2:3b` дала 5/5 на фактах при `temperature 0` и 11/15 на
  15 сэмплах. Прежде чем выбирать модель по числу — повторить прогон.
- После `node:test`-прогонов покрытие считается строго 100% — добавляя ветку, сразу
  добавляй тест на обе её стороны.
