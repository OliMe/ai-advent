# CLAUDE.md

Указания для работы с репозиторием **ai-advent** — набором консольных приложений на
Node.js + TypeScript для работы с LLM через OpenAI-совместимый API (запросы нативным
`fetch`, без сторонних SDK). Несколько независимых пакетов вокруг общего ядра `core`.

## Пакеты

| Пакет | Назначение |
| ----- | ---------- |
| `core/` | Общее ядро: HTTP-клиент к OpenAI-совместимому API, конфиг из окружения, типы, ретраи (429/5xx/сеть) с бэкоффом, `disableThinking`. Плюс движок памяти, сессии, профили, задачи и автономный пайплайн. Остальные пакеты импортируют его напрямую. |
| `llm-cli/` | Консольный чат с моделью: одиночный запрос и интерактивный режим, слоистая память, мультипрофили, задачи и пайплайн (`/run`). **Основной фокус разработки.** |
| `contract-extractor/` | Извлечение реквизитов сторон из договоров (`extract-json`, `list-text`). |
| `contract-generator/` | Генерация синтетических договоров для тестовых данных. |
| `task-solver/` | Решение задачи четырьмя стилями промптинга + оценка GLM. |
| `hf-multi/` | Параллельный вызов нескольких моделей HuggingFace на один промпт. |
| `mcp-probe/` | Песочница-исследование MCP: клиент к любому серверу (stdio/Streamable HTTP, настройка через `.env`), печать списка инструментов или результата вызова инструмента. **Исключение**: использует официальный `@modelcontextprotocol/sdk`. |
| `yandex-ocr-mcp/` | MCP-**сервер** распознавания текста (Yandex Vision OCR): инструмент `recognize-text` (файл/URL/base64), конфиг из `.env`. Два транспорта: stdio (`cli.ts`) и Streamable HTTP (`http.ts`, express + bearer-auth `auth.ts`) для деплоя на VPS (Dockerfile + systemd). **Исключение**: `@modelcontextprotocol/sdk` + `zod` + `express`. Тонкие `server.ts`/`cli.ts`/`http.ts` вне покрытия. |
| `mcp-client/` | Адаптер MCP-серверов к абстракции `ToolSet` ядра: подключение (stdio/HTTP), агрегирование инструментов (неймспейс `сервер__инструмент`), рантайм add/remove. `connectionFactory(onElicit)` — клиент объявляет capability `elicitation` и переадресует серверные запросы `elicitation/create` обработчику приложения (подтверждение операций от сервера). **Исключение**: официальный `@modelcontextprotocol/sdk`; тонкий `connection.ts` вне покрытия. |
| `filesystem-mcp/` | MCP-**сервер** файловой системы с «мягкой» песочницей (allow-list). Инструменты: `read_file`/`write_file`/`append_file`/`list_dir`/`delete_path`. Разрешённые каталоги — позиц. аргументы или `FS_ALLOWED_DIRS`. Путь вне allow-list НЕ отклоняется сразу: через `confirm` (MCP **elicitation**) спрашивается подтверждение пользователя; без обработчика — отказ. Защиты delete: без рекурсии, нельзя удалить корень («стереть всё» невозможно). `classifyPath` (резолв без отказа) + ворота `authorizePath`; `write`/`append` создают родительские каталоги. **Исключение**: `@modelcontextprotocol/sdk` + `zod`; тонкие `server.ts`/`cli.ts` вне покрытия. |
| `scheduler-mcp/` | MCP-**сервер** планировщика отложенных/периодических задач с фоновым исполнением 24/7 (цикл `tick` сам гоняет «созревшие» задачи и копит историю). Инструменты: `schedule_task`/`list_tasks`/`get_task`/`cancel_task`/`pause_task`/`resume_task`/`run_now`/`get_history`. Расписания: `interval`/`daily@HH:MM±TZ`/`once`. **Фаза 1** — исполнители без LLM (`http_check`/`note`), доставка pull через `get_history`; хранилище JSON. **Фаза 2** — исполнитель `agent` (NL-инструкция через `core.Conversation` + встроенные инструменты `get_weather` (wttr.in, без ключа; open-meteo с VPS заблокирован)/`http_get`), доставка `deliver:"telegram"` (best-effort; с VPS Telegram заблокирован → основная доставка через `poll_results` + `llm-cli --watch` с системными уведомлениями). **Фаза 3** — мониторинг: `system_metrics` (метрики VPS + опц. доступность url), `report` (агрегат истории задачи `targetTaskId`). Клиентский `get_my_location` (гео) в llm-cli. Нужен `LLM_*` для агента, `TELEGRAM_*` опц. Два транспорта (stdio/HTTP), деплой на VPS под-путь `/scheduler`. Импортирует `core`. **Исключение**: `@modelcontextprotocol/sdk` + `zod` + `express`; тонкие `cli.ts`/`server.ts`/`http.ts`/`runtime.ts` вне покрытия. |
| `places-mcp/` | MCP-**сервер** поиска организаций рядом. Инструмент `find_places` (text + latitude/longitude + опц. radius/limit; координаты агент берёт из `get_my_location`) — список с расстоянием (гаверсинус, сортировка по близости), адресом, телефоном, часами. **Переключаемые провайдеры** (`PlaceProvider`, env `PLACES_PROVIDER=osm\|yandex`, дефолт **osm**): `provider-osm.ts` — OpenStreetMap Overpass (POST, без ключей; словарь категория→OSM-тег, фолбэк по имени), `provider-yandex.ts` — Yandex Search API (нужен `YANDEX_PLACES_API_KEY`; **закрыт за платный доступ**, оставлен на будущее). Общие типы/утилиты — `geo.ts`. **Исключение**: `@modelcontextprotocol/sdk` + `zod`; тонкие `server.ts`/`cli.ts` вне покрытия. |
| `rag/` | RAG-**индексатор** (День 21). Пайплайн: источник → чанкинг → эмбеддинги → JSON-индекс с метаданными. Источники (автоопределение `detectSource`): локальная папка, GitHub (tarball codeload), URL (обход по ссылкам в пределах origin, глубина-параметр); логика с инъекцией IO, тонкий `node-io.ts` (fs/fetch/tar) вне покрытия. **2 стратегии чанкинга** (`chunkers.ts`): `fixed` (размер+overlap) и `structural` (md по заголовкам, код по файлам; огромный раздел до-резается, сохраняя section). Метаданные на чанк: `source/file/title/section/chunk_id`. Эмбеддинги — `core.EmbeddingsClient` (по умолчанию Ollama, сменяемо через `LLM_EMBEDDINGS_*`). `IndexStore`+`JsonIndexStore` (сменяемо на sqlite/бинарь), `cosine`/`topK`, `computeStats` (сравнение стратегий). CLI `build`/`compare`/`query` (вне покрытия). Импортирует `core`. |
| `rag-mcp/` | MCP-**сервер** RAG (Дни 22–23). Инструмент `search_docs` (query + опц. source/strategy/k + опц. `minScore`/`rerank`/`rewrite`): с `source` (github/папка/URL) индексирует **на лету** при отсутствии индекса (пакет `rag`) и кэширует (`sourceKey` по source+strategy+**схеме эмбеддинга**, каталог `~/.rag-mcp/indexes`), без `source` — по всем кэшированным; возвращает **трассу стадий** «до/после» + фрагменты с метками `[source › file › section]`+score. Плюс `list_indexes`/`build_index`. Ретрив — **staged-конвейер** `retrieve`: (опц. `rewrite`) → эмбеддинг **с префиксом** → `topK(kPre)` → фильтр `minScore` → `rerank` → срез `k`. **День 23 (реранкинг/фильтрация):** префиксы задачи nomic (`search_query: `/`search_document: `, `withPrefix`, в ключе кэша через `embeddingScheme`); порог `RAG_MIN_SCORE`; `rerank` `none`/`mmr` (`mmr.ts`, MMR — релевантность+разнообразие)/`llm` (`rerank-llm.ts`, chat-модель проставляет скоры; интерфейс `RerankProvider` — контракт как у `/rerank` cross-encoder, задел под bge-reranker/TEI); `rewrite` `none`/`expand`/`hyde` (`rewrite.ts`, через chat-модель). Chat-модель для rewrite/LLM-реранка — `RAG_LLM_*` с фолбэком на ядровые `LLM_*` (`loadChatConfig`); нет — фичи выключены, конвейер идёт как есть. Аргументы `search_docs` перекрывают конфиг (сравнение режимов без перезапуска). Стратегия/режимы сменяемы (env/аргумент). Импортирует `rag`+`core` (`EmbeddingsClient`+`ChatCompletionClient`). **Исключение**: `@modelcontextprotocol/sdk`+`zod`; тонкие `server.ts`/`cli.ts`/`runtime.ts` вне покрытия. Подключается к `llm-cli` через `/mcp add` (директива `ragSearchDirective` нацеливает агента). |

Глубоко проработаны `core` и `llm-cli` — остальные пакеты самостоятельны и проще.

## Среда и инструменты

- **Node 24+, нативный TypeScript** — без шага сборки. Файлы запускаются напрямую
  (`node src/cli.ts`), тесты тоже (`node --test '...*.test.ts'`).
- **Импорты — с расширением `.ts`** (`./types.ts`), относительные между пакетами
  (`llm-cli` тянет ядро как `../../core/src/index.ts`). Корневого `package.json`/
  workspaces нет; каждый пакет самостоятелен.
- `tsconfig`: `strict`, `noEmit`, `verbatimModuleSyntax: true`,
  `allowImportingTsExtensions: true`, `module/moduleResolution: NodeNext`. Нет
  `noUnusedLocals`/`noUncheckedIndexedAccess`.
- **Prettier** (в `package.json` каждого пакета): `singleQuote`, `printWidth: 100`,
  `arrowParens: avoid`, `semi: true`, `trailingComma: all`, `tabWidth: 2`.

## Команды (в каждом пакете свои)

```bash
npm --prefix core   run typecheck      # tsc (только проверка типов)
npm --prefix core   test               # node:test + покрытие, порог 100/100/100
npm --prefix core   run write-prettier # форматирование
npm --prefix llm-cli run typecheck
npm --prefix llm-cli test
npm --prefix llm-cli start             # запустить CLI (node src/cli.ts)
```

Конфиг CLI берётся из окружения (`loadConfig`, переменные `LLM_*`, файл `.env`).

## Рабочие соглашения (ОБЯЗАТЕЛЬНЫ)

- **Чистая архитектура во всех пакетах**: SOLID/DRY/KISS, модульность с самого начала,
  переиспользуемая логика — в `core`, специфика CLI — в `llm-cli`. Не сваливать всё в
  один файл (ориентир ~300 строк); тонкий barrel `index.ts`.
- **Имена — полные и описательные** для всех идентификаторов (переменные, функции,
  типы, файлы). Без сокращений вроде `cfg`, `msg`, `idx`.
- **JSDoc — на русском, без указания типов** (типы выражает TypeScript). Поля классов
  объявлять явно.
- **100% покрытие** в каждом пакете: lines/functions/branches. Перед каждым коммитом —
  зелёные `typecheck` и `test`. Ветки `??`/`||`/`?.` и обе стороны тернарника должны
  быть пройдены; избегать мёртвых защит (напр. `arr[0] ?? ''`, когда индекс всегда
  строка) — это недостижимая ветка.
- **Зелёные коммиты**: каждый коммит собирается и проходит тесты сам по себе. Связанные
  изменения объединять, а не дробить ради «по пакету на коммит».
- **Коммит и пуш — раздельно. Пушить ТОЛЬКО по явной просьбе** («Пуш»/«Запушь»). Не
  объединять git-команды в одну. Трейлер коммита:
  `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
- **Живой смоук после реализации**: всегда прогонять настоящий CLI вживую (через
  драйвер с временным `LLM_SESSION_DIR`, флаг `--no-thinking` для GLM), а не только
  мок-тесты. Команды без вызовов модели — бесплатны.
- **Документация — часть реализации**: при добавлении/изменении функционала ОБЯЗАТЕЛЬНО
  дополнять `README.md` (корневой и/или пакета) и `CLAUDE.md` под новое поведение — это
  не отдельная задача «на потом», а часть той же работы.
- **Новые команды → в `/help`**: добавляя интерактивную команду в `llm-cli`
  (реестр в `interactive.ts`), ОБЯЗАТЕЛЬНО внести её в `helpText` (`formatters.ts`) и
  закрепить тестом `helpText`, иначе команда тихо выпадает из справки.
- **Не винить аккаунт/токен пользователя** в ошибках API — сначала диагностировать.
- **`.env` нечитаем** (глобальный deny на чтение). Менять переменные окружения — просить
  пользователя (`! echo 'X=Y' >> path/.env`).

## Архитектура `core`

- `chat-completion-client.ts` — `ChatCompletionClient`: `completeWithUsage` (с
  `AbortSignal`-таймаутом) и `streamWithUsage` (таймаут по простою, дельты через колбэк).
  Ретраи 429/5xx/сети, `disableThinking`.
- `embeddings-client.ts` — `EmbeddingsClient`: OpenAI-совместимый `/embeddings` (батч-вход,
  сортировка по `index`), провайдеро-независим (`EmbeddingsConfig` из `loadEmbeddingsConfig`,
  env `LLM_EMBEDDINGS_URL/MODEL/API_KEY`); `Authorization` только при ключе (Ollama без ключа).
  Для RAG-индексатора (`rag/`). `http-retry.ts` — общие ретраи 429/5xx/сети для обоих клиентов.
- `config.ts` (`loadConfig`/`AppConfig`, `loadEmbeddingsConfig`/`EmbeddingsConfig`), `types.ts`
  (`ChatMessage`, `GenerationLimits`, `Usage`, `ResponseFormat` и т.д.).
- `conversation.ts` — `Conversation`: переиспользуемый раннер диалога над клиентом.
  Держит транскрипт, на `ask(userInput, onDelta?)` шлёт окно истории и возвращает ответ;
  `onDelta` → стрим. На нём строятся интерактив, этапы пайплайна и будущие субагенты.
- `tokens.ts` — оценка токенов, бюджет истории, обрезка скользящим окном, стоимость.
- Сессии: `session.ts` (`Session`, `createSession`, `sessionId`) + `session-store.ts`
  (`FileSessionStore`, атомарная запись). Ветвление/переключение сессий.
- Память (движок): `memory-strategy.ts` (короткая: window/summary/facts),
  `memory-task.ts` (`TaskMemory`), `memory-profile.ts` (`ProfileMemory`),
  `memory-invariants.ts` (`InvariantsMemory` — глобальные жёсткие ограничения),
  `memory-manager.ts` (`MemoryManager` — оркестрирует инварианты+профиль+задачу+короткую,
  подмешивает блоки и директивы персонализации/инвариантов, извлекает память из реплик,
  консолидирует профиль). Хранилища: `profile-store.ts`, `task-store.ts`,
  `invariants-store.ts` (глобальный `~/.llm-cli/invariants.json`).
- Пайплайн задач: `task-run.ts` (`TaskRun`/`Stage`/артефакты/`createRun` + **явный
  автомат жизненного цикла**), `run-store.ts` (`FileRunStore`: `<id>.json` + каталог
  файлов-артефактов), `pipeline-stages.ts` (раннеры этапов + ленивый разбор),
  `pipeline.ts` (`runPipeline` — оркестратор-автомат), `stage-team.ts` (**команда
  агентов на этап**: оркестратор подбирает роли, ограниченно-параллельный веер,
  синтез), `json.ts` (общий ленивый разбор JSON из ответов модели). Подробнее — ниже.
- Явные переходы между состояниями (`task-run.ts`): таблица `ALLOWED_STAGE_TRANSITIONS`
  (рёбра между этапами; перепрыгнуть этап нельзя) + `stagePrerequisiteMet` (есть ли
  артефакт предыдущего этапа: план — только после требований, завершение — только после
  пройденной проверки). `canTransition` сводит их в вердикт, `applyTransition` —
  единственная точка смены этапа: на недопустимом переходе **бросает**
  `InvalidTransitionError` (через неё ходит `transition()` в `pipeline.ts`). `repairStage`
  при возобновлении чинит несогласованный прогон (этап «впереди» своих артефактов —
  правленый/повреждённый файл) откатом назад, не двигая вперёд; драйвер сообщает об этом
  хуком `onStageRepair`. Так ассистент не проскочит этап ни в нормальном ходе, ни после
  паузы.

## Архитектура `llm-cli`

- `cli.ts` → `index.ts` `main()`: разбор аргументов (`args.ts`), режим одного запроса
  (`chat.ts` `runOnce`) или интерактив (`interactive.ts`).
- `interactive.ts` — цикл readline + **реестр команд** (`{matches, run}`, первая
  подходящая по `matches`; порядок важен: точные перед префиксными). Команды: `/help`,
  `/sessions`, `/branch`, `/switch`, `/reset`, `/system`, `/file`, `/temp`, `/task*`,
  `/profile*`, `/run*`, `/mcp*`. Собирает `MemoryManager`, мост к задачам и `RunController`.
- MCP-инструменты агентам (Фаза 3a): `mcp-store.ts` (`FileMcpStore` над `~/.llm-cli/mcp.json`)
  + `McpToolSet` из пакета `mcp-client` собираются в `main()` (если не `--no-mcp`) и передаются
  в `runInteractive`. Инструменты идут **планировщику и исполнителю** (через `ConversationFactory`
  → `StageContext.tools`); вызовы печатаются (`onToolCall`). Команды `/mcp`, `/mcp add <имя>
  <команда|URL>`, `/mcp remove <имя>`, `/mcp reload` — живое подключение/отключение + запись в
  `mcp.json`; серверы подключаются на старте, закрываются на выходе (сбой сервера не валит app).
  Tool-use ядра — в `core` (`ToolSet`, агентный цикл `Conversation`), провайдеро-независимо.
  Чат тоже умеет инструменты (Фаза 3b): при подключённых серверах ход чата идёт агентным
  циклом `completeWithTools` (`chat.ts`) — БЕЗ стрима (tool_calls в потоке сложны), обёрнут в
  контролёр инвариантов; расход токенов суммируется по раундам. Требуется провайдер с
  function-calling (проверено на GLM и DeepSeek). Длина агентного цикла — `AppConfig.maxToolRounds`
  (env `LLM_MAX_TOOL_ROUNDS`, дефолт 12): хватает на длинный кросс-серверный флоу (геолокация →
  места → файл → планировщик). После хода печатается **трасса вызовов** (`formatToolTrace`:
  «🧭 a → b → c») — видно выбор инструментов и порядок маршрутизации по серверам.
- RAG-директива (`rag-directive.ts`, День 22): если среди инструментов есть `…search_docs`
  (подключён `rag-mcp`), в ведущие system-сообщения добавляется `ragSearchDirective` — агент
  вызывает поиск по документам на вопросы о репо/сайте/папке (с source из вопроса) и отвечает
  строго по фрагментам со ссылками на источники. Нет инструмента — директивы нет.
- Подтверждения от серверов (MCP elicitation): `elicitation.ts` — `ElicitationBridge`
  связывает обработчик (создаётся рано, в `main()`, и идёт в `connectionFactory` →
  `McpToolSet`) с запросом к пользователю через readline (`readlineConfirm`), подставляемым в
  `runInteractive`, когда готов ввод. Пока запрос не подставлен — безопасный `decline`. Так
  `filesystem-mcp` может в рантайме спросить разрешение на операцию вне песочницы (⚠ да/нет).
- `chat.ts` — `askModel`/`streamAnswer` (+ спиннер «думает…»), `runOnce`.
- `run-flow.ts` — `RunController` (драйвер `/run`): фабрика диалогов агентов,
  сбор требований, прогон пайплайна с печатью этапов и подтверждением, пауза по Ctrl+C.
- `run-task-bridge.ts` — `MemoryRunBridge` (поверх `MemoryManager` через узкий фасад
  `TaskMemoryFacade`, DIP): связывает прогон с задачей сессии.
- `formatters.ts` (тексты/справка/рендер этапов), `replies.ts` (да/нет/список),
  `paths.ts` (каталоги `sessions`/`tasks`/`profiles`/`runs`), `files.ts`, `errors.ts`,
  `session-flow.ts`.
- `index.ts` — barrel: реэкспортит `core` + локальные модули (тесты импортируют отсюда).

## Модель «задача» (память + пайплайн)

Задача — **единая сущность**, а `/task` и `/run` — два действия над ней: «вести» и
«исполнить».

- `/task [текст]`, `/tasks`, `/task switch|done|delete` — ведение задачи: по ходу диалога
  она копит факты (детали) и подмешивается в контекст чата. Директива персонализации
  велит модели при новой/неоднозначной задаче **сначала задать уточняющие вопросы**, и
  только потом решать.
- `/run [id|описание]` — исполнить задачу пайплайном: без аргумента — текущую, по id —
  существующую, по описанию — новую. Прогон связан с задачей через `taskId`.

Поток `/run`:

1. **Сбор требований** (до планирования): агент-аналитик задаёт уточняющие вопросы по
   одному; ответы пишутся в детали задачи и в транскрипт. Нет вопросов — пропускается.
2. **Фиксированный пайплайн** `planning → execution → verification → completion`
   (пропуск запрещён, каждый этап даёт артефакт). Память задачи (детали + профиль)
   подаётся в планирование и выполнение через `memoryContext`.
3. Провал проверки или отказ на завершении → авто-возврат в `execution` до `maxRetries`
   (по умолчанию 2); исчерпано → пауза. **Завершение требует подтверждения пользователя.**
4. По подтверждению итог дописывается в детали задачи, задача помечается `done`.

Управление прогоном: `/runs`, `/run status [id]`, `/run continue [id]`, `/run edit
<правка>`, `/run abort`. **Ctrl+C** во время прогона ставит его на паузу на границе
этапа (не выходит из приложения). Полный читаемый результат каждого этапа печатается в
консоль и пишется в транскрипт сессии (видно в истории, идёт в контекст чата).
`--ephemeral` держит сессии/задачи/прогоны в памяти, без записи на диск.

Артефакты — структурные поля + `text`. Планирование/проверка/завершение возвращают JSON
(просим **в промпте, без `response_format`** — иначе z.ai/GLM вырезает литерал «json»);
парсер устойчив: берёт объект целиком или первый сбалансированный блок `{…}` из прозы
(`extractJsonObject`), фолбэк — текст/буллеты. **Выполнение возвращает плоский текст** (не
JSON: крупный код ломал экранирование JSON-строки) — `parseExecution` снимает markdown-
ограждение, `summary` = первая строка, `text` = результат; сохраняется файлом (`files[]`).
Планировщик обязан заполнять `steps` (не только `text`); при пустых шагах выполнение и
вывод берут план из `text`. Планировщик добирает пустые критерии повторным запросом
(до 2 раз). Проверяющий сверяет результат по критериям, иначе по шагам, иначе по тексту
плана (без авто-провала на пустых критериях — иначе зацикливается на execution); совсем
пустой план → провал без вызова модели. При возврате к сбору требований устаревшие
замечания проверки очищаются.
Проверяющий работает на низкой температуре (стабильный вердикт). Детали задачи в
контексте агентов ограничены бюджетом токенов (свежие важнее), потолок генерации
этапов — опц. `LLM_STAGE_MAX_TOKENS`.

Инварианты в пайплайне: блок инвариантов подмешивается в контекст ВСЕХ агентов этапов
(аналитик/планировщик/исполнитель/проверяющий/завершающий) — чтобы они не предлагали и не
порождали нарушающее изначально. Перегенерация при нарушении сохраняет формат/полноту ответа.

Инварианты (жёсткий контроль): вывод РЕШАЮЩИХ агентов (чат + этапы planning/execution/
completion, не аналитик/проверяющий) проходит через `enforceInvariants` (`invariant-guard.ts`)
— агент-контролёр (`INVARIANT_CHECKER_SYSTEM`) сверяет результат с инвариантами; при
нарушении называет его и агент перегенерирует (до 2 раз), иначе `InvariantViolationError`
→ в чате отказ, в `/run` пауза этапа. Пустой список инвариантов → контролёр не вызывается.

Команда агентов на этап (`stage-team.ts`): многоагентность спрятана ВНУТРЬ раннера этапа,
автомат и downstream-артефакты не меняются. Поток планирования: оркестратор
(`ORCHESTRATOR_SYSTEM`, концептуально «основной чат») по задаче+памяти решает состав
команды (`orchestrateTeam` → `parseTeamPlan`, зажат по `LLM_MAX_STAGE_AGENTS`, дефолт 4;
`≤1` — многоагентность выключена, оркестратор не зовётся). Простая задача → одна роль →
исходный одиночный `planSolo`. Сложная → роль-эксперты (архитектор/безопасность/QA/…)
работают ограниченно-параллельно (`runRoleExperts`/`mapWithConcurrency`, лимит
`LLM_STAGE_AGENT_CONCURRENCY`, дефолт 2; сбой одной роли пропускается, все упали → откат к
`planSolo`), затем синтезатор сводит вклады в ОДИН `PlanningArtifact` (поле
`contributions`, вклады пишутся файлами `planning-team-*`). Жёсткий `enforce` — на
ИТОГОВОМ синтезированном результате; инварианты идут в контекст всех ролей мягко (через
`memoryContext`). Решение оркестратора печатается хуком `onTeam` (`formatTeam`) только при
команде >1. Ленивый разбор JSON вынесен в `json.ts` (`extractJsonObject`/`parseJsonObject`).

ВАЖНО (качество плана + сходимость). Баланс: слишком строго → проверка не проходит и прогон
зацикливается; слишком мягко → вырожденный план (одна фраза, пустые steps/criteria) проходит
проверку даром. Решения:
- Оркестратор собирает команду на МНОГОГРАННой задаче (несколько аспектов/подсистем дают
  более полный результат), одну роль — для узкой задачи с единственным небольшим результатом.
- Роль-эксперты дают немного критичного, синтезатор/планировщик пишут ДЕТАЛЬНЫЙ план и
  ДИСТИЛЛИРУЮТ критерии в немного (3–6) соразмерных (не объединяя всё подряд).
- Критерии приёмки — главный гейт проверки. Если их нет (модель отдала план прозой), они
  добираются НАПРАВЛЕННЫМ запросом (узкий «выдели 3–6 критериев по плану» модель выполняет
  надёжнее, чем повторную выдачу всего плана JSON-ом). Шаги в `text` допустимы — их разворачивает
  выполнение; форсировать массив `steps` не пытаемся (GLM упирается, добор холостой).
- Проверяющий сверяет ПО СУЩЕСТВУ только заявленные критерии (без выдумывания новых и
  придирок сверх них), мелкие замечания не валят вердикт.
- Наблюдаемость: при команде печатается её состав (`🧩`) и ВКЛАД КАЖДОЙ роли в результате
  планирования (видно, что агенты отработали).
Проверено живыми прогонами задач разной сложности (тривиальная/средняя/широкая): все дают
осмысленные критерии и завершаются автономно.

Не реализовано (на потом): многоагентность на этапе ВЫПОЛНЕНИЯ (механизм `stage-team.ts`
переиспользуем, но пока подключён только к планированию).

## Особенности провайдеров

- **z.ai (GLM)**: использовать `--no-thinking` (`disableThinking`) в смоуках/прогонах,
  иначе модель тратит бюджет на рассуждения; внимательно с массивами `stop` и
  конкурентностью.
- **HuggingFace router**: возможна привязка к провайдеру; `/v1/models` отдаёт
  `context_length` и цены.

## Частые грабли

- `verbatimModuleSyntax: true` → импорты только типов ОБЯЗАНЫ быть `import type`.
- Нет `noUnusedLocals` → неиспользуемые импорты НЕ роняют сборку (но убирать).
- Покрытие ветвей: и `summary` присутствует/пуст, и обе стороны `?:`/`??`/`||` должны
  быть пройдены тестом.
- После `node:test`-прогонов покрытие считается строго 100% — добавляя ветку, сразу
  добавляй тест на обе её стороны.
