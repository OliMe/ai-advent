# Локальная LLM как приватный сервис на своём VPS

Замеры, вердикт по железу и развёрнутый сервис. Всё, что ниже, — измерено, а не оценено;
там, где цифра получена экстраполяцией, это сказано прямо.

## Железо, которое было

| | `узел A` | `узел Б` |
| --- | --- | --- |
| CPU | 4 vCPU AMD EPYC-Milan (AVX2) | 2 vCPU Intel Xeon Silver 4416+ (AVX-512, AMX) |
| RAM | 5.8 ГБ (≈5 свободно) | 3.3 ГБ (≈2.6 свободно) |
| GPU | нет | нет |
| Прочее | реверс-прокси и фоновые сервисы | docker и фоновые сервисы |

Оба узла уже заняты другими сервисами. Это ограничение, а не деталь: инференс не должен отбирать у соседей по узлу CPU.

## Замеры (llama.cpp `llama-bench`, Q4_K_M)

| Модель | `узел A`: промпт | `узел A`: генерация | `узел Б`: генерация |
| ------ | ------------------- | ---------------------- | --------------------- |
| 1.5B | 115 t/s | **40.5 t/s** | 10.0 t/s |
| 3B | 54 t/s | **21.3 t/s** | 4.4 t/s |
| 7B | 23 t/s | **9.8 t/s** | не влезает |

`узел Б` отпадает: 2 vCPU переподписаны соседями по гипервизору. Бэкенд там выбирается
правильный (`libggml-cpu-sapphirerapids.so`, AMX задействован) — дело не в нём.

### Генерация упирается в память, а не в процессор

Скорость генерации × размер весов даёт одну и ту же константу:

| Модель | Генерация | × размер | = |
| ------ | --------- | -------- | - |
| 1.5B | 40.5 t/s | 1.04 ГиБ | **42.1 ГБ/с** |
| 3B | 22.0 t/s | 1.87 ГиБ | **41.2 ГБ/с** |
| 7B | 9.8 t/s | 4.36 ГиБ | **42.7 ГБ/с** |

Чтобы выдать один токен, модель обязана прочитать все свои веса. Полоса памяти этой машины —
~42 ГБ/с, и мы в неё уже упёрлись: эффективность на ядро падает с 6.74 до 5.51 t/s при росте
1→4 потоков. **Больше ядер генерацию не ускорят. Больше RAM не ускорит ничего** — она лишь
позволит запустить модель побольше, которая по той же формуле будет пропорционально медленнее.

Обработка промпта — наоборот, честно линейна по ядрам (14.8 → 27.9 → 40.9 → 52.7 t/s, 89%
эффективности): это перемножение матриц, а не чтение весов.

### Где стена

Промпт на 5652 токена обрабатывается **192 секунды**. Для RAG (3–6к токенов контекста) это
3+ минуты до первого символа. Чтобы получить TTFT ≈ 3 с на 4к токенов, нужно ~1300 t/s против
наших 23–54.

Экстраполяция (*линейное продолжение измеренной кривой — допущение, за пределами 4 ядер не
мерилось*): 16 vCPU → TTFT ~51 с, 64 vCPU → ~13 с, и генерация всё те же ~11 t/s из-за полосы.

**Вывод:** нарастить CPU/RAM у VPS бессмысленно. Для RAG и пайплайна нужен GPU (≥8 ГБ VRAM
под 7B Q4) либо bare-metal с 64–96 ядрами и 12 каналами DDR5. Для короткого чата хватает
того, что есть.

### 7B на `узел A` не помещается

Ollama грузит веса с `mmap = false` и дополнительно репакует их под AVX2 (`CPU_REPACK`, +3 ГБ).
Итого ~4.5 ГБ анонимной памяти при 5.9 ГБ всего. Под лимитом `MemoryMax=4.6G` модель не
загрузилась вовсе (таймаут 7 минут). Рабочая модель на этом узле — 3B.

## Выбор модели: замер, а не интуиция

Две 3B-модели ошибаются по-разному. Батарея из 5 фактических вопросов × 3 сэмпла:

| Модель | Факты | Латиница в ответах | Рецепты: латинских вкраплений |
| ------ | ----- | ------------------ | ----------------------------- |
| `llama3.2:3b` | **11/15** | 0/15 | 8 (плюс иероглифы `切ños`) |
| `qwen2.5:3b` | 7/15 | 3/15 | **1** |

llama знает больше про мир, qwen лучше владеет русским. Обе пишут «Паризь», «Токийо», а llama
однажды выдала «Токيو» арабицей. Температура почти ни при чём (11/15 при 0.3 против 10/15 при
0.8) — дело в самой модели.

Сервис — генератор рецептов, поэтому взята `qwen2.5:3b`: язык важнее фактов, а правильного
ответа у рецепта нет. Замечание на будущее: две 3B разом в память не влезают (2×2.32 ГБ >
`MemoryMax`), и Ollama переключает их за **4.3 с** — вторую модель в сервис даром не добавить.

## Сервис

Развёрнут на `узел A`, см. [`llm-gateway/README.md`](../llm-gateway/README.md).

- **Доступ по сети:** `https://smartnfree.ru/ai/kitchen` (Caddy → `127.0.0.1:3002`).
- **Авторизация:** bearer-токен на все запросы к модели; страница просит его формой.
- **Стабильность при нескольких запросах:** 4 одновременных — 4/4 успешно, строго по очереди
  (5.0 / 9.9 / 14.4 / 18.7 с). Проверено и на уровне Ollama, и через шлюз.
- **Лимиты:** rate limit 10 запросов залпом + 10/мин (11-й → `429`), очередь глубиной 4
  (сверх → `429`), потолок запроса 1500 токенов (сверх → `413`), контекст модели 8192.

### Соседние сервисы не пострадали

Инференсу отданы 3 ядра из 4 (`CPUQuota=300%`, `CPUWeight=20`, `Nice=10`), критичный сосед защищён
(`CPUWeight=200`, `OOMScoreAdjust=-800`), Ollama при нехватке памяти умирает первой
(`OOMScoreAdjust=800`). Под полной нагрузкой: ping 31.6 → 32.2 мс (разброс ±1.7), round-trip
ssh не изменился, `mpstat` показывает 75% в категории `nice` и 24% idle — одно ядро всегда
свободно.

## Что осталось за кадром

- Наш `llm-cli` в grounded-режиме на этом VPS **непригоден** (3+ минуты на ход). На M2 с
  `qwen2.5:7b` тот же ход — ~40 с. Домашняя машина обыгрывает VPS впятеро на обработке промпта.
- Экстраполяция по числу ядер не проверена живьём — единственное недоказанное место в этом
  документе.
