# rag-mcp

MCP-**сервер** RAG: инструмент `search_docs` ищет релевантные фрагменты в проиндексированных
документах, **строя индекс на лету**, если его ещё нет. Подключается к `llm-cli` через `/mcp add` —
подключён → у агента появляется способность отвечать по документам; не подключён → обычный чат.

> Импортирует `rag` + `core` (`EmbeddingsClient`). Сознательно использует официальный
> `@modelcontextprotocol/sdk` + `zod` (как другие MCP-серверы). 100% покрытие логики; тонкие
> `server.ts`/`cli.ts`/`runtime.ts` вне покрытия.

## Инструменты

| Инструмент | Назначение |
| ---------- | ---------- |
| `search_docs` | Поиск фрагментов по запросу. `query` + опц. `source` (github-url / путь / URL доков) + опц. `strategy` (`fixed`/`structural`) + опц. `k`. Есть `source` без индекса → **строит на лету** и кэширует; без `source` → ищет по всем кэшированным индексам. Возвращает фрагменты с метками `[source › file › section]` и оценкой близости. |
| `list_indexes` | Перечисляет кэшированные индексы (источник, стратегия, число чанков). |
| `build_index` | Заранее индексирует источник (опц. `strategy`) — чтобы первый поиск был быстрым. |

## Как работает

1. **Поиск релевантных чанков:** запрос эмбеддится той же моделью, что и чанки (по умолчанию
   Ollama `nomic-embed-text`), считается косинусная близость к векторам чанков, берётся top-`k`.
   Конвейер staged: `retrieve(kPre)` → стадия `rerank/фильтр` (**пока проходная — хук Дня 23**) →
   срез `k`.
2. **Индексация на лету:** при отсутствии индекса для `source` он строится (загрузка через пакет
   `rag`: папка / GitHub-tarball / веб-обход → чанкинг → эмбеддинги) и кэшируется на диск. Ключ
   кэша учитывает **стратегию**, поэтому индексы разных стратегий сосуществуют.
3. **Ответ модели:** `search_docs` возвращает фрагменты как результат вызова; агентный цикл
   `llm-cli` кладёт их в контекст, и модель отвечает по ним, ссылаясь на источники (директива
   `ragSearchDirective` в `llm-cli` нацеливает на это).

## Настройка

По умолчанию эмбеддинги идут в локальный Ollama — поднимите его:

```bash
ollama serve
ollama pull nomic-embed-text
```

Переменные окружения (все опциональны):

| Переменная | Назначение | По умолчанию |
| ---------- | ---------- | ------------ |
| `RAG_STRATEGY` | Стратегия чанкинга (`fixed`/`structural`) | `structural` |
| `RAG_TOP_K` | Сколько фрагментов отдавать | `5` |
| `RAG_TOP_K_PRE` | Сколько доставать до стадии rerank/фильтра (хук Дня 23) | `= RAG_TOP_K` |
| `RAG_CACHE_DIR` | Каталог кэша индексов | `~/.rag-mcp/indexes` |
| `RAG_CRAWL_DEPTH` | Глубина веб-обхода | `2` |
| `RAG_CHUNK_SIZE` / `RAG_CHUNK_OVERLAP` / `RAG_MAX_SECTION` | Параметры чанкинга | `2000` / `256` / `2000` |
| `LLM_EMBEDDINGS_URL` / `LLM_EMBEDDINGS_MODEL` / `LLM_EMBEDDINGS_API_KEY` | Эмбеддинги (сменить Ollama на удалёнку) | Ollama-дефолты |

## Подключение к `llm-cli`

```
/mcp add rag node /путь/к/rag-mcp/src/cli.ts
```

Дальше просто спрашивайте про репозиторий/сайт/папку — агент сам вызовет `search_docs` с нужным
источником, при необходимости построит индекс и ответит по документации со ссылкой на источники.
Пример: «Что делает репозиторий https://github.com/krutoo/utils?» → индекс на лету → ответ по README.

## Структура

| Файл | Назначение |
| ---- | ---------- |
| `src/config.ts` | `loadRagConfig` (стратегия, k/kPre, чанкинг, эмбеддинги). |
| `src/cache-key.ts` | `sourceKey(source, strategy)` — ключ кэша. |
| `src/retrieval.ts` | Staged-конвейер поиска (retrieve → rerank-хук → срез k). |
| `src/index-cache.ts` | `ensureIndex` — из кэша или сборка на лету (инъекция IO). |
| `src/format.ts` | Форматирование результатов и списка индексов. |
| `src/tools.ts` | Обработчики `search_docs`/`list_indexes`/`build_index`. |
| `src/runtime.ts` / `src/server.ts` / `src/cli.ts` | Боевые зависимости + MCP-сервер + stdio (вне покрытия). |
