# rag-mcp

MCP-**сервер** RAG: инструмент `search_docs` ищет релевантные фрагменты в проиндексированных
документах, **строя индекс на лету**, если его ещё нет. Подключается к `llm-cli` через `/mcp add` —
подключён → у агента появляется способность отвечать по документам; не подключён → обычный чат.

> Импортирует `rag` + `core` (`EmbeddingsClient`). Сознательно использует официальный
> `@modelcontextprotocol/sdk` + `zod` (как другие MCP-серверы). 100% покрытие логики; тонкие
> `server.ts`/`cli.ts`/`runtime.ts` вне покрытия.

## Инструменты

| Инструмент | Назначение |
| ---------- | ---------- |
| `search_docs` | Поиск фрагментов по запросу. `query` + опц. `source` (github-url / путь / URL доков) + опц. `strategy` (`fixed`/`structural`) + опц. `k`. **Плюс переопределения качества (День 23):** `minScore` (порог косинуса 0..1), `rerank` (`none`/`mmr`/`llm`), `rewrite` (`none`/`expand`/`hyde`). Есть `source` без индекса → **строит на лету** и кэширует; без `source` → ищет по всем кэшированным. Возвращает **трассу стадий** (`🔎 кандидатов N → порог≥X: M → rerank(mode): K`) и фрагменты с метками `[source › file › section]` + оценкой. |
| `list_indexes` | Перечисляет кэшированные индексы (источник, стратегия, число чанков). |
| `build_index` | Заранее индексирует источник (опц. `strategy`) — чтобы первый поиск был быстрым. |

## Как работает

Конвейер `retrieve` — staged, каждая стадия наблюдаема через трассу «до/после»:

```
(опц. rewrite запроса) → эмбеддинг с префиксом → topK(kPre) → фильтр minScore → rerank → срез k
```

1. **Переписывание запроса (rewrite, День 23, опц.):** `expand` дописывает к запросу синонимы и
   смежные термины; `hyde` генерирует гипотетический документ-ответ и эмбеддит **его** (HyDE —
   часто ближе к искомым чанкам, чем короткий вопрос). Обе — через chat-модель; по умолчанию `none`.
2. **Префиксы задачи nomic (День 23):** запрос и документ эмбеддятся с разными префиксами
   (`search_query: ` / `search_document: `) — модель различает «вопрос» и «пассаж», точность выше.
   Пустая строка в `RAG_QUERY_PREFIX`/`RAG_DOC_PREFIX` отключает их (для моделей без префиксов).
3. **Фильтр по порогу (minScore, День 23):** кандидаты ниже порога косинуса отсекаются (`0` — выкл).
4. **Переранжирование (rerank, День 23):** `mmr` — Maximal Marginal Relevance (релевантность +
   разнообразие, штраф почти-дублей; дефолт); `llm` — chat-модель проставляет скоры релевантности
   (задел под настоящий cross-encoder — интерфейс `RerankProvider`, контракт как у `/rerank`).
5. **Индексация на лету:** нет индекса для `source` → строится (папка / GitHub-tarball / веб-обход →
   чанкинг → эмбеддинги) и кэшируется. Ключ кэша учитывает стратегию **и «схему» эмбеддинга**
   (модель + префиксы), поэтому индексы разных стратегий/схем сосуществуют и не переиспользуются
   с несовместимой схемой.
6. **Ответ модели:** `search_docs` отдаёт фрагменты + трассу; агентный цикл `llm-cli` кладёт их в
   контекст, модель отвечает по ним со ссылкой на источники (директива `ragSearchDirective`).

### rewrite/LLM-реранк — отдельная chat-модель

Эти стадии требуют chat-модель. Она берётся из `RAG_LLM_*`, а при их отсутствии — фолбэком из
ядровых `LLM_*`. Ни того, ни другого нет → rewrite/LLM-реранк просто выключены (остальное работает).
Можно завести отдельную «лёгкую» модель под rewrite/rerank, не трогая основную.

### Сравнение качества (демо на 8 вопросах по README пакетов репозитория)

| Режим | hit@1 | hit@3 |
| ----- | ----- | ----- |
| baseline (День-22: без префиксов, `rerank=none`) | 1/8 | 3/8 |
| +префиксы +MMR +порог | 1/8 | 5/8 |
| +HyDE +LLM-реранк (полный) | **6/8** | **7/8** |

Bi-encoder-улучшения (префиксы/MMR/порог) поднимают **recall** (в top-3 попадает больше нужного);
точность top-1 чинит именно **LLM-реранк** — он разгребает выдачу, где длинный документ забивал её
своими чанками. Порядок стадий и переключение режимов через аргументы `search_docs` позволяют
сравнивать «до/после» без перезапуска сервера.

## Настройка

По умолчанию эмбеддинги идут в локальный Ollama — поднимите его:

```bash
ollama serve
ollama pull nomic-embed-text
```

Конфигурация читается из **`.env` рядом с пакетом** (`rag-mcp/.env`) — по пути модуля, а НЕ из
текущего рабочего каталога. Поэтому конфиг одинаков, откуда бы сервер ни запустили, в том числе
когда его поднимает `llm-cli` как MCP-процесс (у того свой cwd). Уже заданные переменные окружения
имеют приоритет над файлом (`loadEnvFile` их не перезаписывает), так что `env`-блок записи сервера в
`~/.llm-cli/mcp.json` при желании перекрывает файл. Готовый шаблон со всеми переменными и дефолтами —
[`.env.example`](.env.example) (`cp .env.example .env`). Переменные окружения (все опциональны):

| Переменная | Назначение | По умолчанию |
| ---------- | ---------- | ------------ |
| `RAG_STRATEGY` | Стратегия чанкинга (`fixed`/`structural`) | `structural` |
| `RAG_TOP_K` | Сколько фрагментов отдавать (итог после rerank) | `5` |
| `RAG_TOP_K_PRE` | Сколько доставать до стадии rerank/фильтра (kPre ≥ k) | `20` |
| `RAG_MIN_SCORE` | Порог косинуса на стадии фильтра (0..1; `0` — выкл) | `0` |
| `RAG_RERANK` | Режим переранжирования (`none`/`mmr`/`llm`) | `mmr` |
| `RAG_MMR_LAMBDA` | Баланс релевантность/разнообразие MMR (0..1) | `0.7` |
| `RAG_REWRITE` | Переписывание запроса (`none`/`expand`/`hyde`) | `none` |
| `RAG_QUERY_PREFIX` / `RAG_DOC_PREFIX` | Префиксы задачи nomic (пусто — отключить) | `search_query: ` / `search_document: ` |
| `RAG_CACHE_DIR` | Каталог кэша индексов | `~/.rag-mcp/indexes` |
| `RAG_CRAWL_DEPTH` | Глубина веб-обхода | `2` |
| `RAG_CHUNK_SIZE` / `RAG_CHUNK_OVERLAP` / `RAG_MAX_SECTION` | Параметры чанкинга | `2000` / `256` / `2000` |
| `LLM_EMBEDDINGS_URL` / `LLM_EMBEDDINGS_MODEL` / `LLM_EMBEDDINGS_API_KEY` | Эмбеддинги (сменить Ollama на удалёнку) | Ollama-дефолты |
| `RAG_LLM_URL` / `RAG_LLM_MODEL` / `RAG_LLM_API_KEY` | Chat-модель для rewrite/LLM-реранка | фолбэк на `LLM_*` |
| `RAG_LLM_TEMPERATURE` / `RAG_LLM_NO_THINKING` | Температура (дефолт 0.2) / отключить «рассуждения» (GLM) | `0.2` / выкл |

## Подключение к `llm-cli`

```
/mcp add rag node /путь/к/rag-mcp/src/cli.ts
```

Дальше просто спрашивайте про репозиторий/сайт/папку — агент сам вызовет `search_docs` с нужным
источником, при необходимости построит индекс и ответит по документации со ссылкой на источники.
Пример: «Что делает репозиторий https://github.com/krutoo/utils?» → индекс на лету → ответ по README.

Стадии Day-23 (`rerank`/`rewrite`/`minScore`) агент задаёт **дефолтами сервера** (из `rag-mcp/.env`)
или **аргументами** вызова `search_docs`. Чтобы включить rewrite/LLM-реранк по умолчанию, положите в
`rag-mcp/.env` chat-модель и режимы:

```bash
printf 'RAG_RERANK=llm\nRAG_REWRITE=hyde\nRAG_MIN_SCORE=0.5\nRAG_LLM_NO_THINKING=1\n' >> rag-mcp/.env
# chat-модель: свои RAG_LLM_* или ядровые LLM_* (URL/MODEL/API_KEY)
```

Демонстрация «до/после» в чате: задайте один вопрос при `RAG_RERANK=none RAG_REWRITE=none` (базовый
режим), затем поменяйте на `llm`/`hyde` и `/mcp reload` — сравните ответ и процитированный источник.
Либо в одном сеансе переключайте режим прямо в запросе («…вызови search_docs с rerank=llm, rewrite=hyde»).

## Структура

| Файл | Назначение |
| ---- | ---------- |
| `src/config.ts` | `loadRagConfig` (стратегия, k/kPre, порог, rerank/rewrite, префиксы, эмбеддинги) + `loadChatConfig` (RAG_LLM_* → фолбэк LLM_*) + `embeddingScheme`. |
| `src/env.ts` | `loadPackageEnv`/`packageEnvPath` — загрузка `.env` рядом с пакетом (по пути модуля, не из cwd). |
| `src/cache-key.ts` | `sourceKey(source, strategy, scheme)` — ключ кэша со «схемой» эмбеддинга. |
| `src/prefix.ts` | `withPrefix(embed, prefix)` — префиксы задачи nomic для эмбеддера. |
| `src/retrieval.ts` | Staged-конвейер (rewrite → эмбеддинг → topK → фильтр → rerank → срез k) + трасса. |
| `src/mmr.ts` | `mmrRerank` — MMR-переранжирование (релевантность + разнообразие). |
| `src/rewrite.ts` | `makeRewriter` — переписывание запроса (expand/HyDE) через chat-модель. |
| `src/rerank-llm.ts` | `RerankProvider` + chat-реализация + `makeLlmReranker` (задел под cross-encoder). |
| `src/index-cache.ts` | `ensureIndex` — из кэша или сборка на лету (инъекция IO). |
| `src/format.ts` | Форматирование результатов, трассы стадий и списка индексов. |
| `src/tools.ts` | Обработчики `search_docs`/`list_indexes`/`build_index` + переопределения аргументами. |
| `src/runtime.ts` / `src/server.ts` / `src/cli.ts` | Боевые зависимости + MCP-сервер + stdio (вне покрытия). |
